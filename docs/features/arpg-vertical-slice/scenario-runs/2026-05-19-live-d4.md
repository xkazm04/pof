## Findings for sub-project D4 (iteration)

### Critical finding: the harness has zero idempotency

**All three steps fast-failed in 7.9 seconds, including Step 8 ih-1 which passed in D3.** This is a regression, but informative — D3's success poisoned D4's run.

### Root cause

When D3's Step 8 ih-1 ran successfully:
1. Real Claude Code created the UE5 assets.
2. `waitForCliComplete` detected the session finished with success=true.
3. `useModuleCLI.onComplete(true)` fired (per `useModuleCLI.ts:97`).
4. The checklist item was auto-marked complete in `useModuleStore.checklistProgress`.
5. The progress state was persisted to `~/.pof/pof.db` via the Zustand `persist` middleware.

When D4 ran the same spec with a fresh Playwright browser context:
- New browser context → fresh localStorage, fresh IndexedDB.
- BUT the PoF dev server (PID 50132, alive across D2/D3/D4) reads from the same `~/.pof/pof.db`.
- On page load, the persisted checklist progress is re-fetched.
- ih-1 is loaded as `checked=true`.
- Per `RoadmapChecklist.tsx:648`: the Claude button only renders `{!checked && !isActive && (<AccentButton...>Claude</AccentButton>)}`. With `checked=true`, no button.
- The harness's `claudeBtn.count()` returns 0 → fails with "No 'Claude' button inside ih-1 row (Cards layout); hover may not have triggered the visibility change."

The error message is misleading — hover did trigger, but the button literally doesn't exist in the DOM for a completed item.

### What this means for D5+ (mandatory before any further live work)

**D5's first task must be idempotency.** Options:

1. **Reset DB state before each test** — call `DELETE FROM checklist_progress WHERE module_id IN (...)` via a test fixture or beforeAll hook. Cleanest; requires a server-side endpoint OR direct DB access from the spec via `better-sqlite3` (same package PoF uses).
2. **Use uncompleted items per run** — instead of always testing ih-1, scan for the first uncompleted item in the target module and use that. Resilient; doesn't require DB access; but tests *whatever's available* rather than specific items.
3. **Disable persistence in test mode** — add a query param like `?test=true` that PoF detects and uses an in-memory store. Most invasive; touches PoF source.
4. **Manually clear `~/.pof/pof.db` between runs** — `rm ~/.pof/pof.db` before each invocation. Crudest; loses non-test state too (project lists, build profiles, etc.).

Recommended D5 approach: **Option 1** (server-side reset via a test-only API route OR direct SQL). It's the cleanest separation.
