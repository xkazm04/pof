# HUD Sub-Project Findings ‚Äî Vertical-Slice Combat HUD

**Date:** 2026-05-22
**Repos:** UE project `github.com/xkazm04/pof-exp`; docs in the PoF app repo.
**Outcome:** **Delivered** ‚Äî the slice shows a player health bar and a target/enemy health bar, built entirely in C++, with gameplay provably intact.

---

## What was built

Two C++ classes in the UE project's `Source/PoF/UI/`, plus a wiring script:

| File | Class | Role |
|---|---|---|
| `VSHUDWidget.h/.cpp` | `UVSHUDWidget : UUserWidget` | Builds its widget tree in C++ (no UMG editor, no Widget Blueprint): a player health bar (top-left, green) and a target/enemy health bar (top-centre, red), each bound to a GAS `UAbilitySystemComponent`'s `Health`/`MaxHealth`. |
| `VSHUD.h/.cpp` | `AVSHUD : AHUD` | Creates the widget, adds it to the viewport, and binds the player + the enemy (a retry timer handles late-spawning ASCs). |
| `Content/Python/wire_hud.py` | ‚Äî | Sets `HUDClass = /Script/PoF.VSHUD` on `/Game/VerticalSlice/BP_VSGameMode`. |

No UMG-editor assets and no Widget Blueprints ‚Äî the project's `BindWidget`-coupled
HUD widgets (`ARPGHUDWidget` etc.) were deliberately not used; they remain for a
future UMG-editor pass.

## Verification found and fixed a real bug

The first verification run showed `AVSHUD` running (`[VSHUD]` log lines) but the
two health bars not visible on screen. A diagnostic established:

- **Root cause:** `UVSHUDWidget` built its widget tree in `NativeConstruct()`.
  For a `UUserWidget`, the Slate tree is constructed in `RebuildWidget()` /
  `Initialize()`, which runs **before** `NativeConstruct()` ‚Äî so assigning
  `WidgetTree->RootWidget` in `NativeConstruct` had no effect and the widget was
  empty. (The `UBossHealthBarWidget` "reference" cited during implementation is
  dead code ‚Äî never instantiated ‚Äî so its pattern was never actually proven.)
- An earlier guess that `MaxHealth = 0` (from the missing `AttributeInitTable`)
  was **wrong** ‚Äî a temporary log confirmed the binds seed `Health = 100,
  MaxHealth = 100` (the `UARPGAttributeSet` constructor defaults), consistent
  with the functional test reading the enemy's GAS Health as 100.
- The top-left `Lv / XP / HP / MP / Stamina` text on screen is **engine debug
  text** (`GEngine->AddOnScreenDebugMessage` from
  `ARPGPlayerCharacter::UpdateDebugDisplay` and `ARPGCharacterBase`'s tick,
  `#if !UE_BUILD_SHIPPING`) ‚Äî not a HUD widget. It was left untouched.

**Fix** (commit `dc25e4a`): move the tree construction into a `RebuildWidget()`
override; give each `UProgressBar` an explicit style (solid dark track + bright
fill) inside a `UBorder` frame so a bar is visible at any fill level;
reposition the player bar below the debug-text rows; `AddToViewport(30)` for
Z-order.

## Final result

- **Gemini visual check (real-window launch):** confirms a **green player
  health bar top-left** and a separate **red target/enemy health bar
  top-centre**, the latter labelled `Enemy` with `100 / 100` text ‚Äî both filled
  and clearly visible over the 3D scene.
- **PS-1 functional test:** re-run green ‚Äî #2 movement, #3 attack, #4 damage,
  #5 death + loot all pass, `Result={Success}`. The HUD does not affect
  gameplay.
- Minor, non-blocking: the engine debug text pins to the top-left corner and
  overlaps the player bar slightly; the player bar is still distinct beneath
  it, and the enemy bar is clean. The debug text is `#if !UE_BUILD_SHIPPING`
  and would not appear in a Shipping build.
