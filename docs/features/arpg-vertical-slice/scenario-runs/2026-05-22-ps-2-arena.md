## Fix Loop (Step 2)

The functional test passed on the **first** run on the rebuilt arena — the new geometry,
collision, and actor placement did not break the gameplay loop. **No gameplay fix was
needed.** Collision (`CTF_USE_COMPLEX_AS_SIMPLE`) and placement (actors above the floor,
inside the walls) were correct as built by the prior PS-2 tasks.

One **rendering** defect was found and fixed during the Step 3 visual check (it did not
affect the headless `-nullrhi` functional test, which passed regardless):

### Fix 1 — Arena rendered pitch black: lights were not Movable + un-baked static lighting

**Symptom:** A `-game` screenshot of the rebuilt level showed an all-black world with a
working HUD and the on-screen warning **"LIGHTING NEEDS TO BE REBUILT (1 unbuilt object)"**.

**Root cause:** `SM_Arena` is STATIC mobility and this headless pipeline never runs
Lightmass. The `DirectionalLight` was STATIONARY and the `SkyLight` STATIONARY — static
geometry with no baked lightmaps and no Movable light renders black.

**Fix:** Added a section to `build_arena_ue.py` (Section 5, after the actor reposition)
that sets the `DirectionalLight` and `SkyLight` components to **Movable** mobility, so
the arena is lit fully dynamically with no bake step.

### Fix 2 — Arena still dark: DirectionalLight grazing the floor

**Symptom:** After Fix 1 the "rebuild lighting" warning was gone, but the arena was still
near-black in the screenshot.

**Root cause:** The `DirectionalLight` sat at its default rotation (0, 0, 0) — pointing
horizontally along +X, grazing the floor at ≈ 0°, so the floor received almost no light.

**Fix:** Extended the same `build_arena_ue.py` section to angle the `DirectionalLight`
down (rotation pitch **-50°**, yaw -35°) and set a healthy intensity (DirectionalLight
`intensity = 6.0`, SkyLight `intensity = 3.0`, SkyLight real-time capture on). After this
the arena renders fully lit (see Gemini check below).

Each fix: edited `build_arena_ue.py`, kept the archived copy
(`docs/features/arpg-vertical-slice/ps-2-artifacts/build_arena_ue.py`) in sync, re-ran the
script via the full editor (`UnrealEditor.exe ... -ExecutePythonScript=... -unattended`),
and re-ran the functional test — it stayed GREEN on all of #2–#5 after both fixes.

## Texturing — What Was and Was Not Achieved

**Materials: fully achieved and verified.** Diagnostic Python on the live asset confirmed:

- All 9 PolyHaven JPGs imported as `T_*` Texture2D assets (normals as `TC_NORMALMAP`).
- `M_Arena_{Floor,Wall,Pillar}` each built with the albedo TextureSample connected to
  Base Color (and normal/rough samples), `BLEND_OPAQUE`.
- `SM_Arena`'s three slots (`M_Floor`/`M_Wall`/`M_Pillar`) resolve to those materials.
- The **spawned arena actor's StaticMeshComponent reports all 3 `M_Arena_*` materials
  bound at runtime** — `num_materials=3`, no null/override slots.

So the real PolyHaven asphalt/concrete textures are imported, wired, assigned, and bound
on the live actor — the texturing pipeline is correct end-to-end.

**Surface polish: partial.** In the screenshot the surfaces read as a regular
light/dark grid rather than crisp asphalt/concrete. The cause is **UV quality, not
missing materials**: the Blender arena uses gray-box cube-projection UVs
(`uv.cube_project`, 2 m cube size), which tile the 1k textures ~10× across the 20 m
floor and seam at every cube face. At the grazing in-game camera angle and the windowed
capture resolution this tiled, seamed texture reads grid-like. The materials and textures
themselves are correct; what is missing is artist-quality UV layout / texture scaling.
This is acceptable for PS-2 (an environment-blockout sub-project) and is a clean
follow-up: re-UV the arena parts (planar/box-unwrap with sane texel density) so the
PolyHaven detail reads as a surface rather than a grid.

## Technical Notes

- The headless functional test runs with `-nullrhi`, so it is unaffected by the lighting
  defect — the rendering fixes were needed only for the visual sanity check, and the
  test passed GREEN before, between, and after them.
- `HighResShot` via `-ExecCmds` fires during engine startup before any frame renders and
  produces no usable file (or a black/early frame). Reliable capture: launch
  `UnrealEditor.exe` in `-game -windowed`, let the map settle ~40 s, then grab the window.
- `MaterialEditingLibrary.get_material_expressions` does not exist in UE 5.7 Python;
  `get_material_property_input_node(material, MP_BASE_COLOR)` was used instead to confirm
  the Base Color wiring.
- Re-running `build_arena_ue.py` must go through the full editor
  (`UnrealEditor.exe ... -ExecutePythonScript=... -unattended`) — the lightweight
  `-run=pythonscript` commandlet crashes on UE 5.7's Interchange FBX import.
