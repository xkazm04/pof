## D1 stub-run surprises (real findings — must address in D2)

### Finding 1: Zero CLI prompts captured despite 7 dispatch-triggering steps passing

**Symptom:** Steps 8-14 all clicked module checklist "Run with Claude" buttons; spec reports each as ✅ pass. But `Captured dispatches: 0 CLI prompt dispatches`.

**Root cause (per implementer's diagnosis):** `useModuleCLI.execute()` at `src/hooks/useModuleCLI.ts:140-155` awaits `useProjectStore.getState().scanProject()` before calling `sendPrompt()`. In the headless test env, `/pof/status` returns `Failed to fetch` (visible in dev-server logs as `[PoF-Bridge] GET /pof/status failed: Failed to fetch`). The async chain likely throws or hangs silently before reaching `dispatchEvent('pof-cli-prompt')`. The capture-phase listener installed by `setupHarnessMode()` is correct in principle — it just never receives an event because the event never fires.

**D2 mitigation options:**
- (a) **Seed project context** via `page.evaluate` writing to `useProjectStore` before the dispatch-triggering steps. Most invasive but most realistic.
- (b) **`page.route()` mock for `/pof/status`** returning a synthetic project context payload. Lower-impact; doesn't require touching the store.
- (c) **Bypass `scanProject` by calling `sendPrompt(prompt)` directly** via a page-side helper — sidesteps the issue but means the harness doesn't exercise the real `execute()` path.

Recommended: (b) for D2 stub mode (keeps stub mode hermetic); (a) for D2 live mode (real project context needed anyway).

### Finding 2: Zero cook executes captured because no packaging profile exists in test DB

**Symptom:** Step 19 hit the skip path with note "No existing profile cards; create-profile flow not in D1 scope."

**Root cause:** PoF's packaging panel renders Add-Platform buttons when there are zero profiles, and Profile cards (with the `pof-module-packaging-start-cook-{profileId}` testId) only when at least one profile exists. The test runs against a clean `~/.pof/pof.db` so no profiles exist → no cards → no start-cook button → cook stub never fires.

**D2 mitigation:** Add a pre-test seeding step that POSTs a default Win64 Shipping profile via `/api/packaging/profiles` before reaching Step 19. Sub-project A noted this endpoint exists at `src/app/api/packaging/profiles/route.ts:9-75`. Alternatively, click the Add-Platform → fill-and-save flow within the test itself (more realistic, more brittle).

### Finding 3: SidebarL1 toggle behavior bit the spec; helper added

**Symptom (caught + fixed by implementer):** `SidebarL1.handleClick` at `src/components/layout/SidebarL1.tsx:15-21` *toggles* a category: clicking the active L1 category collapses the L2 list. Naively re-clicking `pof-sidebar-nav-item-core-engine` between sibling sub-module visits (Step 10 → Step 11) collapsed the sidebar and broke L2 navigation.

**Fix in place:** `navigateToSubModule(page, categoryId, l2TestId)` helper in the spec checks L2 visibility before deciding whether to (re-)click L1.

**D2 / future implication:** Any subagent extending this spec must use the helper, not raw `getByTestId('pof-sidebar-nav-item-X').click()`. Consider documenting this pattern in the spec's header comment.

### Finding 4: `test.skip()` inside `test.step()` aborts the entire test

**Symptom (caught + fixed by implementer):** The plan's literal `test.skip(condition, msg)` inside the Phase 8 `test.step()` skipped the whole test (Playwright's documented behavior). The findings doc never wrote because tear-down was bypassed.

**Fix in place:** Phase 8 now uses a conditional `if (mode !== 'live')` block that records 3 skip results for Steps 22-24; the whole test body is wrapped in `try/finally` so `writeFindings()` always runs.

**D2 implication:** Anyone editing the spec must preserve the try/finally + manual skip-recording pattern for Phase 8.
