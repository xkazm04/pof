## Part C — arena re-texture via the 3D-texture endpoint ⚠️ (endpoint unusable)

- **`export_arena_obj.py`** → `Content/ArenaBuild/Arena.obj` via
  `StaticMeshExporterOBJ`. UV-mapped: 1668 verts / 1668 UVs / 556 faces; face
  lines are `f v/vt` triplets. **Works.**
- **3D-texture endpoint — NOT usable.** Live API discovery (2026-05-23):
  - `POST /models-3d/upload` takes **`modelExtension: "obj"`** (not `modelType`),
    and returns an **S3 presigned multipart POST** (`modelUrl` + `modelFields`
    JSON), **not** a PUT. The OBJ upload **works** (S3 returns 204; the model is
    retrievable at `GET /models-3d/{id}` with a `meshUrl`).
  - **`POST /generations-texture` returns 404** ("Endpoint not found"). Probed
    ~12 candidate create paths; `/models-3d/texture` etc. only resolve as
    `/models-3d/{id}` (GET, DELETE). **There is no public endpoint to GENERATE a
    PBR texture set on an uploaded model.** The capability the
    `leonardo-api-capabilities` memory recorded as "legacy-flagged but live" has
    been removed from the public REST API — only 3D model upload + retrieval +
    deletion remain.
  - **`generateTextureOn3DModel` corrected** to the real upload flow (it uploads
    successfully, then throws a clear error at the 404 create step, and deletes
    the uploaded model on cleanup). All test models created during discovery were
    `DELETE`d (account left clean).
- **Outcome (honest fallback per spec):** no 3D PBR set was produced, so
  `retexture_arena_3d.py` was not written/run and the Gemini A/B was N/A. The
  arena **keeps PS-3's textures**, now served through `M_ARPG_Surface_Master` via
  the `MI_Arena_*` instances (Part B). Parts A + B ship full value.

## Part C alternative — Scenario PBR prototype (Option B) ✅ built, awaiting key

Research (May 2026) confirmed Leonardo **removed** 3D texture generation: live
`/generations-texture` 404s with both `modelId` and the documented `modelAssetId`,
and the current official SDK (`@leonardo-ai/sdk` v4.21.3, repo pushed 2026-05-23)
exposes only `uploadModelAsset` + `delete3DModelById` under `ThreeDModelAssets` —
no `createTextureGeneration`. The blog/docs that still describe it are stale.

**Scenario.gg** still offers true PBR-set generation, so it was prototyped as a
drop-in for the removed Leonardo capability:

- **`src/lib/scenario.ts`** — `generateTexture({ prompt, modelId?, width?, height? })`.
  Flow: Basic auth (base64 `KEY:SECRET`) → `POST /generate/txt2img-texture` → poll
  `GET /jobs/{jobId}` until `status:"success"` → `metadata.assetIds` → `GET
  /assets/{id}` → `asset.url`. Returns the full map set + convenience
  `albedoUrl`/`normalUrl`/`roughnessUrl`/`metallicUrl`/`heightUrl`/`aoUrl`.
  `classifyPbrMap()` buckets each asset by type hint.
- **`src/app/api/scenario/route.ts`** — `POST /api/scenario { prompt, modelId?, … }`
  → the PBR set.
- **Tests:** 9 vitest (mocked) — Basic-auth header, request body shape, job poll,
  asset-URL resolution, PBR classification, failure + missing-key paths.

**To enable (live):**
1. Add `SCENARIO_API_KEY` + `SCENARIO_API_SECRET` to the app `.env` (gitignored).
2. `POST /api/scenario { "prompt": "dark fantasy dungeon stone, seamless PBR",
   "modelId": "<a Scenario texture model id>" }`.
3. Confirm against the live API (no key was available at build time): the
   `txt2img-texture` body field names (`modelId`/`width`/`height`), and the asset
   type-hint field used by `classifyPbrMap`/`extractAssetTypeHint`. All three are
   isolated single-line tweaks in `scenario.ts`.

**Wiring into the arena re-texture:** point `retexture-arena-runner.mjs` (or a new
runner) at `/api/scenario` instead of the Leonardo `texture3d` mode, download the
returned maps to `Content/ArenaBuild/textures_3d/`, then run `retexture_arena_3d.py`
(already designed in the plan) to import them + apply `MI_Arena_3D`. Scenario's 2D
seamless PBR set pairs naturally with branch #5's world-aligned UVs.
