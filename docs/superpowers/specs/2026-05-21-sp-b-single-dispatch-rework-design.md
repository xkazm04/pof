# SP-B Single-Dispatch Rework + Layer-6 Fix

## Context

SP-B's gameplay-chain live run has failed **four** times, each on a different
defect, each correctly fixed, each revealing the next layer. The defects all
sit in PoF's CLI-session subsystem and surface specifically when the autonomous
harness drives **back-to-back same-module dispatches in one browser page**:

1. dispatch race / re-dispatch double-submit → stuck `isRunning`;
2. cold-start window too short → dispatch mis-read as failed;
3. `CompactTerminal` auto-submit unreliable + abnormal-exit not releasing
   `isRunning`;
4. (run #4) auto-submit fixed — acb-1 ran a real 60 s session to "Done" — but
   `onTaskComplete` is deferred behind a callback-POST promise (`cbPromise`),
   which did not resolve, so `isRunning` stuck again and the next same-module
   step (acb-4) hung on its disabled "Claude" button.

**What is proven:** a *single isolated dispatch* works — acb-1 has run clean
three separate times (64 s / 54 s / 60 s). **What never worked in 4 attempts:**
*chaining* same-module dispatches in one page.

The user chose: fix the run-#4 defect ("layer 6"), then change SP-B's execution
model to single-dispatch-per-step so the chained-dispatch failure class is
structurally eliminated.

## Goals

1. **Layer-6 fix:** `onTaskComplete` (and the `session.isRunning` release) must
   always fire within a bounded time after a session ends — never gated
   indefinitely on a callback POST.
2. **SP-B single-dispatch rework:** every gameplay dispatch runs in its own
   isolated Playwright page, so no chained-`isRunning` collision is possible and
   no single test can hang the whole run.
3. Keep all 10 operator steps (8 gameplay dispatches + 2 informational).

## Non-goals

- **No further CLI-subsystem rewrite.** Layer-6 is a targeted ~5-line fix.
- **No new gameplay scope.** Same 8 dispatch items + 2 informational steps.
- **No removal of the harness Send-click workaround** — it stays as
  defense-in-depth (idempotent-safe; CLI Fix #1 is the real fix).

## Decision record (from brainstorming)

1. **Path:** fix layer 6, then switch SP-B to single-dispatch-per-step (chosen
   over single-dispatch-only-no-fix, and over wrapping the initiative).
2. **Steps 15-16 kept** — feature-matrix scan + evaluator deep-eval remain as
   2 informational isolated runs.
3. **Structure:** one isolated `test()` per step, generated from a `STEPS`
   array via a loop (vs. 10 hand-written blocks — rejected, not DRY).

## Design

### Part A — Layer-6 fix (`src/components/cli/useTaskQueue.ts`)

The `result` SSE handler (`useTaskQueue.ts:310-338`) currently does:

```ts
const cbPromise = cbMatch ? resolveCallback(...).then(...) : Promise.resolve();
cbPromise.finally(() => {
  const tid = currentTaskIdRef.current;
  if (tid) { registerTaskComplete(tid, instanceId, !data.isError); onTaskComplete?.(tid, !data.isError); }
});
```

`onTaskComplete` — which releases `session.isRunning` — fires only when
`cbPromise` settles. If `resolveCallback`'s POST hangs, `isRunning` is stuck
forever (run #4).

**Fix:** race `cbPromise` against a bounded timeout so the completion always
runs. The callback POST still gets its chance to finish first (the intent —
"data in the DB before onTaskComplete" — is preserved for the common fast
case), but a hung POST can no longer strand the session:

```ts
const settleBound = Promise.race([
  cbPromise,
  new Promise<void>((resolve) => setTimeout(resolve, UI_TIMEOUTS.callbackSettleMax)),
]);
settleBound.finally(() => {
  const tid = currentTaskIdRef.current;
  if (tid) { registerTaskComplete(tid, instanceId, !data.isError); onTaskComplete?.(tid, !data.isError); }
});
```

`UI_TIMEOUTS.callbackSettleMax` is a new constant (~10 s — generous for a
healthy callback POST, bounded for a hung one). Exact placement and whether
`cbPromise` already has a `.catch` (so the race can't reject) are settled during
planning by reading the full handler.

This is the same class as the CLI-subsystem spec's Fix #2 (a completion path
must not depend on something that can hang) — completing the set.

### Part B — SP-B single-dispatch rework (`e2e/arpg-vertical-slice-sp-b.spec.ts`)

Replace the two chunk `test()` blocks (which chained 3 and 5 steps in one page)
with **one `test()` per step**, generated by looping over a `STEPS` array.

```ts
interface SpbStep {
  /** Test title — includes a "chunk 1"/"chunk 2" tag for --grep grouping. */
  title: string;
  kind: 'dispatch' | 'feature-matrix' | 'evaluator';
  target?: RoadmapDispatchTarget;     // for kind 'dispatch'
  liveExpectations?: ArtifactExpectation[];  // for kind 'dispatch'
}

const STEPS: SpbStep[] = [ /* acb-1, acb-4, ae-2, al-5, al-6, au-1, au-2, au-7,
                              feature-matrix, evaluator — in dependency order */ ];

for (const step of STEPS) {
  test(step.title, async ({ page }) => {
    test.setTimeout(12 * 60_000);
    const harness = await setupHarnessMode(page);
    try {
      await runLiveStep(harness, page, step.title, async () => { /* dispatch + verify */ });
    } finally {
      await harness.writeFindings({ filenameSuffix: `sp-b-${slug(step.title)}` });
    }
  });
}
```

Key properties:

- **Isolation.** Playwright gives each `test()` a fresh browser context/page by
  default. Each dispatch is the only session on its page — no chained
  `isRunning` collision is possible, and a stuck `isRunning` from one step
  cannot affect the next (fresh page).
- **Bounded, non-hanging.** Each test has its own `~12 min` `test.setTimeout`.
  A failed or stuck step fails *only itself* at its own cap; Playwright then
  runs the next test. There is no single 40-minute test that can consume the
  whole run on one hang.
- **Dependency order.** `STEPS` is ordered acb-1 → acb-4 → ae-2 → al-5 → al-6 →
  au-1 → au-2 → au-7 → feature-matrix → evaluator. Playwright runs file tests
  in order with `workers: 1` (the existing config), so later steps build on the
  artifacts of earlier ones.
- **`chunk 1` / `chunk 2` tags** in test titles let the controller `--grep`
  the combat+enemy foundation (chunk 1: acb-1/acb-4/ae-2), inspect its UE
  artifacts, then run chunk 2 — preserving the artifact-inspection checkpoint.
- **Per-step findings.** Each test calls `writeFindings` with a per-step
  `filenameSuffix`, producing one small findings file per step. The controller
  aggregates them into one SP-B findings doc after a run.
- **Stub + live.** Unchanged duality: stub mode short-circuits
  `waitForCliComplete`; the stub run proves all 10 tests' wiring. The 8 dispatch
  steps use the existing `dispatchRoadmapChecklistItem` (Send-click workaround
  retained); steps 15-16 keep their inline feature-matrix/evaluator logic.

The `runLiveStep` / `dispatchStep` / artifact-verification helpers from the
current SP-B spec are reused; only the test *structure* changes (10 isolated
tests instead of 2 chained chunks).

## Definition of done

1. Layer-6 fix: `useTaskQueue`'s `result` handler completes the task within a
   bounded time even if the callback POST hangs; `npm run validate` green.
2. `arpg-vertical-slice-sp-b.spec.ts` reworked into 10 isolated per-step
   `test()` blocks via the `STEPS` loop; stub run passes all 10.
3. Gated live: chunk 1 (acb-1/acb-4/ae-2 — isolated) run; acb-1 and acb-4 both
   complete with no chained hang; artifacts inspected; then chunk 2.
4. Per-step findings aggregated into one SP-B findings doc; committed.
5. Chat summary.

**Success criterion:** SP-B's gameplay steps run to completion as isolated
dispatches with no chained-`isRunning` hang — producing the combat / enemy /
loot / HUD UE artifacts the vertical slice needs.

## Risks & mitigations

- **Per-test `completeSetupWizard` overhead** (~10-15 s × 10). Accepted — it is
  the price of isolation; total overhead ~2 min.
- **A step still fails on its own** (e.g. an environmental `claude.cmd` glitch,
  bug #3). Mitigation: isolation means it fails *only itself* and records a
  fail; the rest of the run proceeds. Re-running one step is one short run.
- **Layer-6 timeout too short for a slow-but-healthy callback POST.** Mitigation:
  ~10 s is generous; it is a ceiling — a fast POST still settles first. The
  constant can be raised without code change.
- **Findings aggregation.** Per-step files are simple, robust artifacts; the
  controller aggregates. No shared-state machinery across tests.
