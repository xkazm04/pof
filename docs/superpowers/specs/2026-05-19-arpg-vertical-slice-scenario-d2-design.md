# Sub-project D2: ARPG vertical-slice — harness fixes + first live attempt

## Context

D1 produced the scenario harness (`e2e/arpg-vertical-slice.spec.ts`) and ran it once in stub mode. The findings doc surfaced four real issues:

1. **0 CLI prompts captured.** `useModuleCLI.execute()` awaits `scanProject()` which silently fails on `/pof/status` fetch in test env. The dispatch never fires.
2. **0 cook executes captured.** Packaging panel requires at least one profile to render the start-cook button; clean test DB has zero.
3. SidebarL1 toggle bug — already fixed in D1 by `navigateToSubModule` helper.
4. `test.skip()` inside `test.step()` aborts the whole test — already fixed in D1.

D2 addresses findings 1 + 2 and makes a **first, small live-mode attempt** to validate the live dispatch wiring works end-to-end. The live attempt is scoped to two steps only — Step 6 (build verify) and Step 8 ih-1 (Input Actions) — both chosen for fast completion + low blast radius.

The full 24-step live run remains deferred to D3 (which will iterate based on D2's live findings).

## Goals

1. **Close D1 finding 1**: Add `/pof/status` mock in stub mode so `scanProject()` resolves and CLI dispatches fire.
2. **Close D1 finding 2**: Add packaging-profile seed helper; call before Step 19 so the cook flow can trigger.
3. **Verify the fixes**: regenerate `2026-05-19-stub.md`; dispatch counts must be CLI > 0 + cook ≥ 1.
4. **Add live-mode completion-wait helpers**: `waitForCliComplete(page, sessionLabel, timeoutMs)` and `waitForCookComplete(page, timeoutMs)` with 10-min default timeout and clean session-abort on exceed.
5. **Create a thin D2 live spec** that exercises Step 6 + Step 8 ih-1 with `HARNESS_MODE=live`.
6. **Run the D2 live spec once** and commit the findings doc — regardless of pass/fail. The job is to find out.

export interface SeedProfileOptions {
  name?: string;
  platform?: 'Win64' | 'Linux' | 'Mac';
  config?: 'Shipping' | 'Development' | 'DebugGame';

export async function seedPackagingProfile(
  page: Page,
  opts: SeedProfileOptions = {},
): Promise<string> {
  const profile = {
    name: opts.name ?? 'Win64-Shipping-Harness-Seed',
    platform: opts.platform ?? 'Win64',
    config: opts.config ?? 'Shipping',
    // Other required BuildProfile fields filled with defaults; shape verified at write time
    // by reading src/lib/packaging/build-profiles.ts.

  const res = await page.request.post('/api/packaging/profiles', { data: profile });
  if (!res.ok()) {
    throw new Error(`seedPackagingProfile: POST /api/packaging/profiles returned ${res.status()}`);
  }
  const body = await res.json();
  // Response envelope shape: { success: true, data: { id, ... } }
  return body?.data?.id ?? '';
}

### Phase 2 — Live-wait helpers

New exports in `harness-mode.ts`:

```typescript
export interface WaitResult {
  success: boolean;
  durationMs: number;
  timedOut: boolean;
  /** Captured terminal output excerpt (live mode only; empty in stub). */
  outputExcerpt?: string;
}

/**
 * Wait until the CLI session for the given module/sessionKey transitions
 * from running → stopped, OR timeoutMs elapses.
 *
 * Stub mode: short-circuits to {success: true, durationMs: ~100, timedOut: false}.
 *
 * Live mode: polls DOM for the pof-cli-panel-running-indicator testId
 * (which is present iff isRunning=true on the active session). When it
 * disappears, considers the session done. Reads the final isRunning + a
 * notion of success from a follow-up DOM check or store evaluation.
 *
 * On timeout: dispatches a click on the abort/Stop button (if present)
 * OR injects a setSessionRunning(false, false) call via page.evaluate
 * if the store is reachable. Records timedOut: true.
 */
export async function waitForCliComplete(
  page: Page,
  sessionLabel: string,
  timeoutMs: number = 600_000,
): Promise<WaitResult>;

/**
 * Wait until pof-cook-progress-result testId is visible, OR timeoutMs.
 * Stub mode: synthetic SSE in harness-mode.ts finishes in <2s, so this
 * effectively just waits for the result element + returns success=true.
 */
export async function waitForCookComplete(
  page: Page,
  timeoutMs: number = 600_000,
): Promise<WaitResult>;
```

Implementation choices:

- **Live polling strategy is DOM-based** by default. The `pof-cli-panel-running-indicator` testId (added in sub-project B) is present iff a session is running. Polling for absence is the readiness signal.
- **Fallback strategy:** if the running indicator's behaviour proves unreliable (e.g., the indicator stays mounted but invisible), fall back to `page.evaluate(() => useCLIPanelStore.getState().sessions[].isRunning)`. The store must be globally accessible; check if PoF exposes it on window in dev mode. If not, the fallback requires a PoF source change — which is out of scope. Document the limitation in findings.
- **Abort on timeout:** try DOM-first (click the Stop button if visible — TerminalInput renders one when isStreaming=true). If no Stop button, use `page.evaluate` to call `useCLIPanelStore.getState().setSessionRunning(id, false, false)` if the store is reachable.

### Phase 3 — D2 live attempt

`e2e/arpg-vertical-slice-live-d2.spec.ts`:

```typescript
import { test, expect } from '@playwright/test';
import { stat } from 'node:fs/promises';
import { join } from 'node:path';
import { setupHarnessMode, waitForCliComplete } from './helpers/harness-mode';

const PROJECT_PATH = 'C:\\Users\\kazda\\Documents\\Unreal Projects\\PoF';
const STEP_TIMEOUT_MS = 600_000; // 10 min per spec decision

async function enterWorkspace(page: any): Promise<void> {
  // Same shape as the main spec.
}

test.describe('ARPG vertical slice — D2 live attempt', () => {
  test.setTimeout(30 * 60_000); // 30-min total budget

  test('attempts Step 6 + Step 8 ih-1 in live mode', async ({ page }) => {
    expect(process.env.HARNESS_MODE, 'D2 live spec requires HARNESS_MODE=live').toBe('live');
    const harness = await setupHarnessMode(page);

    // Step 6 — Verify build (no code generation; just runs UE5 build).
    await test.step('Step 6 (LIVE): Verify build', async () => {
      await enterWorkspace(page);
      await page.getByTestId('pof-sidebar-nav-item-project-setup').click();
      // SetupWizard is in the default-active category; find + click the build-verify button.
      const verifyBtn = page.getByTestId('pof-setup-wizard-build-verify-btn');
      await verifyBtn.click();
      const result = await waitForCliComplete(page, 'build-verify', STEP_TIMEOUT_MS);
      harness.recordStepResult({
        step: 'Step 6 (LIVE): Verify build',
        status: result.timedOut ? 'fail' : result.success ? 'pass' : 'fail',
        durationMs: result.durationMs,
        notes: result.timedOut ? `Timed out after ${STEP_TIMEOUT_MS}ms` : (result.outputExcerpt ?? ''),
      });
    });

    // Step 8 ih-1 — Input Actions (creates IA_Move + IA_Attack data assets).
    await test.step('Step 8 ih-1 (LIVE): Input Actions', async () => {
      await page.getByTestId('pof-sidebar-nav-item-game-systems').click();
      await page.getByTestId('pof-sidebar-l2-nav-item-input-handling').click();
      await page.getByRole('tab', { name: 'Roadmap' }).click();
      const ih1 = page.getByTestId('pof-module-input-handling-checklist-item-ih-1');
      const playBtn = ih1.locator('button').filter({ hasText: /Run|Claude|Play/i }).first();
      await playBtn.click();
      const result = await waitForCliComplete(page, 'input-handling-ih-1', STEP_TIMEOUT_MS);

      // Verify expected UE5 artifacts (paths approximate; actual paths captured in findings).
      let artifactCheck = '';
      try {
        await stat(join(PROJECT_PATH, 'Content', 'Input', 'Actions', 'IA_Move.uasset'));
        await stat(join(PROJECT_PATH, 'Content', 'Input', 'Actions', 'IA_Attack.uasset'));
        artifactCheck = 'IA_Move.uasset + IA_Attack.uasset both exist.';
      } catch (err) {
        artifactCheck = `Artifact check failed: ${err instanceof Error ? err.message : String(err)}`;
      }

      harness.recordStepResult({
        step: 'Step 8 ih-1 (LIVE): Input Actions',
        status: result.timedOut ? 'fail' : result.success ? 'pass' : 'fail',
        durationMs: result.durationMs,
        notes: `${result.timedOut ? 'Timed out. ' : ''}${result.outputExcerpt ?? ''}\nArtifact check: ${artifactCheck}`,
      });
    });

    await harness.writeFindings(); // Writes to scenario-runs/{date}-live-d2.md (mode = live).
  });
});
```

## Risks & mitigations

- **`/pof/status` mock shape wrong.** Mitigation: implementation verifies the expected response shape by reading `src/stores/projectStore.ts` + the PoF-Bridge fetch site. If the mock is incomplete, rerun stub spec to see if `scanProject()` resolves; iterate as needed.
- **Live-wait helper can't see the running indicator.** Mitigation: documented fallback to store-access via `page.evaluate`. If the store isn't globally accessible, the live run logs that as a finding and times out by walltime instead of session-state.
- **Live run never returns.** Mitigation: `test.setTimeout(30 * 60_000)` upper bound + per-step 10-min cap. Worst case: Playwright times the whole test out at 30 min and partial findings are not written. Mitigation-of-mitigation: wrap the test body in try/finally so `writeFindings()` runs even on test-level timeout (though Playwright may interrupt anyway).
- **Live run modifies the UE5 project at `C:\Users\kazda\Documents\Unreal Projects\PoF`.** This is BY DESIGN — that's what live mode does. The project is NOT a git repo (per project memory), so we can't easily roll back. User should be aware that ih-1 (Input Actions) will write real files under `Content/Input/Actions/`. Document this in the findings doc.
- **Claude Code not authenticated.** Mitigation: live run will fail fast (dispatch hangs); findings doc records the failure pattern; D3 explicitly requires `claude --version` working before attempting live.
- **Packaging-profile seed leaks across runs.** Mitigation: each run creates a new profile with a timestamp-suffixed name; old profiles accumulate harmlessly. Optional cleanup helper in a future sub-project.
