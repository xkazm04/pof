# UI Perfectionist — Evaluator UI Dashboards

> Context: Evaluator UI Dashboards (Quality & Evaluation)
> Files read: 22 (all 22 evaluator/*.tsx in scope, plus 1 cross-check on src/components/ui/StatStrip.tsx)
> Total: 11 — Critical: 1, High: 5, Medium: 4, Low: 1

> Note on the brief's premise: It calls out "each one rolls its own KPI card / score badge / status chip". This is verifiable — I count 11 ad-hoc `StatCard`/`MetricCard` definitions across the suite (`AssetCodeOracle`, `AssetScout`, `Combat`, `Aggregate`, `CrossModuleOverlap`, `Economy`, `Pattern`, `PromptEvolution`, `Perf`, `PostProcessStudio`, `SessionAnalytics`, `WeeklyDigest`) plus `MiniBar`, `MiniSparkline`, `StatusRow`, `SignalCard` siblings. A `StatStrip` primitive *already exists* in `src/components/ui/` and is not used by any of the 22 dashboards. The duplication is real and addressable. Findings 1, 2, 3, and 6 are direct consequences.

## 1. Eleven local `StatCard`/`MetricCard` reimplementations across the dashboard suite

- **Severity**: Critical
- **Category**: Component Architecture / Design System
- **File**: src/components/modules/evaluator/AggregateQualityDashboard.tsx:773-805 · CrossModuleOverlapPanel.tsx:241-265 · AssetCodeOracleView.tsx:256 · AssetScoutView.tsx:206 · CombatSimulatorView.tsx:555 · EconomySimulatorView.tsx:645 · PatternLibraryView.tsx:287 · PromptEvolutionView.tsx:1187 · PerformanceProfilingView.tsx:621 · PostProcessStudioView.tsx:822 · SessionAnalyticsDashboard.tsx:170 · WeeklyDigestView.tsx:323
- **Scenario**: Twelve files in this scope each define a private `function StatCard(...)` or `function MetricCard(...)` that renders the same recipe: small icon + uppercase 2xs label + bold value + optional sub-line + accent color. The implementations diverge in subtle ways: AggregateQualityDashboard uses `text-lg font-bold` for the value, AssetScout uses `text-2xl`, CrossModuleOverlap omits the sub-line entirely, Combat wraps in `motion.div` with stagger, others don't. `src/components/ui/StatStrip.tsx` already exists with a tested sparkline-aware implementation, but **zero** of these 12 files import it. Each new dashboard ships a 13th–14th variant.
- **Root cause**: The suite has no `KPICard` primitive (StatStrip is a horizontal strip, not a 4-up grid card), and no policy that "evaluator dashboards must use shared metric components." Authors copy-paste from the previous file they shipped.
- **Impact**: Visual KPI cards drift in 4-up rows that sit side-by-side in the Overview tab — value font sizes, gap rhythm, motion staggering, and "0" empty-state colors all differ. A user who scans Quality → Health → Roadmap sees three different "stat card" silhouettes and learns to distrust them as a comparable signal. Every new view ships another StatCard. Adds ~40 LOC of dead drift per file.
- **Fix sketch**: Promote a `<KPICard icon label value sub accent variant?='solid'|'soft' size?='sm'|'md' />` primitive into `src/components/ui/`, modelled on StatStrip's color/sparkline conventions. Migrate the 12 call sites in one wave (the prop shape is identical across all 12, only the styling diverges). Add an ESLint rule or a comment in the primitive: "Do not redefine MetricCard locally — extend KPICard." Same for `<StatusRow>` (currently 1 definition in AggregateQualityDashboard, but the pattern reappears as inline JSX in HolisticHealth's `ModuleHeatCell` and UnifiedSummary's `MiniBar`).

## 2. Each dashboard hand-rolls its own page header (icon-tile + title + sub + action)

- **Severity**: High
- **Category**: Visual Consistency / Component Architecture
- **File**: src/components/modules/evaluator/AssetScoutView.tsx:91-100 · PerformanceProfilingView.tsx:95-104 · PatternLibraryView.tsx:108-117 · CombatSimulatorView.tsx:118-127 · EconomySimulatorView.tsx:102-111 · HolisticHealthView.tsx:103-114 · CalendarRoadmapView.tsx:238-247 · GameDesignDocView.tsx (similar) · CodebaseArcheologistView.tsx (similar)
- **Scenario**: Five views share the *exact* gradient-tile recipe `w-10 h-10 rounded-xl bg-gradient-to-br from-{color}-500/20 to-{color}-500/20 border border-{color}-500/30 flex items-center justify-center` with only the color tokens swapped — emerald→cyan (Asset Scout), rose→orange (Perf), violet→blue (Patterns), red→orange (Combat), amber→orange (Economy). HolisticHealth and Calendar use a different recipe: bare `w-10 h-10 rounded-lg bg-{color}-500/10` (no gradient, no border). EvaluatorModule's main header (line 64-67) uses neither — just a bare `<Radar className="w-6 h-6 text-[#ef4444]" />` next to the title. Three header conventions sit one tab-switch apart.
- **Root cause**: No `<DashboardHeader icon title subtitle accent action />` primitive. Each new view picks whichever sibling looked nicest at copy-paste time.
- **Impact**: A user navigating Quality → Health → Patterns → Perf sees three different visual treatments of "I am the page title." The accent color of the tile is sometimes the module color (Asset Scout = emerald, the marketplace accent), sometimes the dashboard family color (Perf = rose, but the evaluator family color is `MODULE_COLORS.evaluator` red). The header bar — usually the most-glanced element — communicates inconsistent brand grammar.
- **Fix sketch**: Extract `<DashboardHeader icon title subtitle accent action />` to `src/components/ui/`. The accent prop drives the gradient tile; the action slot takes a refresh button or single primary CTA. Standardize on the gradient variant (it has a measurable weight; the bare-tile variant in HolisticHealth/Calendar reads as second-class). Migrate all 9+ headers; remove the bespoke `bg-emerald-500/10`, `bg-blue-500/10`, etc. tile blocks. Same change unifies the action-button styling — currently each view styles its primary CTA differently (`bg-emerald-500/10 text-emerald-400 hover:bg-emerald-500/20` in HolisticHealth vs `bg-rose-500/10 text-rose-400 hover:bg-rose-500/20` in Perf vs `bg-[#ef4444]/10 text-[#ef4444] border border-[#ef4444]/20` in AggregateQuality).

## 3. Tailwind named colors fight CSS-token colors — same pixel ends up two ways

- **Severity**: High
- **Category**: Design System
- **File**: src/components/modules/evaluator/HolisticHealthView.tsx:91-93,118,128,239-241,350-356 · CrashAnalyzerView.tsx:35-39 · PerformanceProfilingView.tsx:32-37 · CodebaseArcheologistView.tsx:44-48 · PatternLibraryView.tsx:27-31 · AggregateQualityDashboard.tsx:381,386,473-475 · GDDComplianceView.tsx:59,71,97
- **Scenario**: Status colors flow through *two parallel pipelines* in this codebase: (a) the canonical token export `STATUS_SUCCESS / STATUS_WARNING / STATUS_ERROR / ACCENT_EMERALD` from `@/lib/chart-colors`, used in `style={{ color: STATUS_ERROR }}`, and (b) Tailwind `text-emerald-400 / text-red-400 / text-amber-400` named utilities applied via className. Within a single file (`HolisticHealthView`) you'll see `style={{ color: STATUS_ERROR }}` on one line and `text-red-400` on the next, plus literal `text-[#f87171]` (AggregateQuality 381,386) and `text-[#ef4444]` (Compliance 59,71,97). Three sources of truth for "warning red" exist and they don't render identical pixels — `STATUS_ERROR` (`#dc2626`-ish from the token) ≠ `text-red-400` (`#f87171`) ≠ `text-[#f87171]`. The Crash Analyzer file is the most extreme: its `SEVERITY_STYLE` table uses Tailwind classes (`bg-red-400/10 border-red-400/20 text-red-400`) but the `SEVERITY_PILLS` array two functions later uses token constants (`STATUS_ERROR, STATUS_BLOCKER`). Same severity rendered with two different palettes in the same view.
- **Root cause**: No lint rule or convention forbids Tailwind color utilities. `chart-colors.ts` is the design-token registry, but it isn't enforced; the `text-emerald-400`/`bg-rose-500/10` shortcut is faster to write so authors use it.
- **Impact**: Critical state colors are not visually consistent. A "critical" severity badge in CrashAnalyzer reads slightly different from a "critical" severity badge in DeepEvalResults (which uses `STATUS_ERROR`) sitting one tab over. Theme switching cannot be implemented coherently — flipping a CSS variable will only re-tint half the surface. 274 occurrences of Tailwind named colors across 17 files is the scale.
- **Fix sketch**: Treat `chart-colors.ts` as the sole color source. Add a `<SeverityIcon severity="critical|high|medium|low" />` and `<SeverityChip>` primitive that internalizes the token mapping; ban Tailwind named colors at the lint level for `evaluator/*.tsx`. Replace the 5 `SEVERITY_CONFIG` / `SEVERITY_STYLE` / `PRIORITY_STYLE` per-file maps with one shared `SEVERITY_TOKENS` exported from `chart-colors.ts`. (Bonus: the Tailwind-class severity tables also pre-bake bg + border + text strings — consolidating yields one source of opacity convention rather than five.)

## 4. Three hand-rolled SVG chart libraries with three different axis/grid conventions

- **Severity**: High
- **Category**: Visual Consistency / Component Architecture
- **File**: src/components/modules/evaluator/HolisticHealthView.tsx:531-687 (BarChartSimple, LineChartSimple, AreaChartSimple, BurndownChart) · AggregateQualityDashboard.tsx:833-872 (MiniSparkline) and 874-948 (TrendChart) · UnifiedSummaryView.tsx:345-404 (CombinedHealthGauge — radial) · CalendarRoadmapView.tsx:393-612 (SVG Gantt) · ProjectHealthDashboard.tsx:54-68 (radar polar helpers)
- **Scenario**: Five files each ship 50–200 lines of bespoke SVG chart code. They diverge on every primitive choice: HolisticHealth's `LineChartSimple` uses `viewBox="0 0 ${data.length * 60} ${h}"` with `preserveAspectRatio="none"` — points are positioned at `i * 60 + 30`; AggregateQuality's `TrendChart` uses fixed pixel `w=160` with no viewBox stretching; CalendarRoadmap renders absolute-positioned text at integer fontSize 9, 10, 11 in three different places (lines 422, 444, 471, 483). Grid line opacity: HolisticHealth uses `border-border/30`, AggregateQuality uses `stroke="var(--border)" strokeWidth="0.5"`. Today/reference markers: CalendarRoadmap uses `MODULE_COLORS.setup` for "today" (a *blue* — not the time-token), HolisticHealth uses `STATUS_NEUTRAL` dashed for the ideal line, AggregateQuality has none.
- **Root cause**: There is no `recharts` (or shared abstraction) in scope. Each author picks raw SVG and re-derives axis math. The d3-style `viewBox` decision (stretch vs. fixed) was made independently four times.
- **Impact**: The Calendar Gantt (the most visible "roadmap" view) shares zero primitives with the Holistic burndown — same data shape (week-bucketed completion), three different visual languages. A user comparing milestone progress on the burndown to deadline variance on the Gantt cannot mentally line up the axes. Cosmetic upkeep (changing grid color, adding hover tooltips) requires editing all five files; in practice it never happens — that's why Calendar has no hover tooltips at all (line 452-459: title attributes only) while AggregateQuality's TrendChart has SVG `<title>` per data point.
- **Fix sketch**: Either (a) introduce `recharts` (already a project precedent in the wider stack — verify) and migrate the four time-series charts (`LineChartSimple`, `AreaChartSimple`, `BurndownChart`, `TrendChart`) to it, or (b) extract `<LineChart>`, `<AreaChart>`, `<BurndownChart>` primitives in `src/components/ui/charts/` with a single grid/axis/today-marker convention. Keep `MiniSparkline` (the inline 48×16 variant) as a thin wrapper over the same primitive. The Gantt is special enough to remain bespoke, but it should at least share the today-marker color (`STATUS_INFO` or a dedicated `--marker-today` token), and its SVG `<text>` font sizes should derive from a 3-tier scale rather than ad-hoc 9/10/11.

## 5. CrossModuleOverlapPanel duplicates DependencyGraph node code & DependencyGraph/NexusView duplicate each other

- **Severity**: High
- **Category**: Component Architecture
- **File**: src/components/modules/evaluator/DependencyGraph.tsx:33-63 vs. NexusView.tsx:26-54 · CrossModuleOverlapPanel.tsx:267-283 (ModuleBubble) vs. DependencyGraph.tsx (graph nodes)
- **Scenario**: `DependencyGraph` and `NexusView` share an identical `MODULE_POSITIONS` table (4 cols × 3 rows of `arpg-*` modules), an identical `getNodeCenter` helper, and the same node-rendering pattern with size constants only off by ±20px (`COL_WIDTH 180/200`, `NODE_W 140/160`). The two files are 583 + 919 lines and the first ~200 lines of each are mirror copies. `CrossModuleOverlapPanel`'s `ModuleBubble` (line 267-283) reimplements the same "module-as-node" surface — same inline rgba stack — but in pill form with `rgba(248, 113, 113, ${intensity * 0.15})` literally hard-coded (the only place in the suite that uses raw rgba; elsewhere it goes through `withOpacity`). The brief's claim is verified: "cross-module overlap panel duplicates dependency-graph node code."
- **Root cause**: NexusView was clearly written by copying DependencyGraph and adding layers; the layout primitives never moved into a shared file. ModuleBubble in OverlapPanel was written independently because its color is fixed (`rgba(248,113,113,...)`) rather than inherited from the per-module color, so the author forked the recipe.
- **Impact**: Three views render "modules in the ARPG topology" with three different size and color conventions. When the team adds `arpg-cinematics` to the registry, three files need updates; in practice only one will be updated and the topology will fall out of sync silently. NexusView's COL_WIDTH=200 means its nodes drift further apart than DependencyGraph's COL_WIDTH=180 — same data, different visual proximity, a real usability cost when reading "which modules cluster together."
- **Fix sketch**: Extract `src/components/modules/evaluator/_shared/ModuleTopology.ts` exporting `MODULE_POSITIONS`, `getNodeCenter`, layout constants, and a `<ModuleNode>` JSX primitive with a `variant="graph" | "nexus" | "bubble"` discriminator (pseudo-3D vs flat vs pill). DependencyGraph and NexusView consume the primitive; OverlapPanel's ModuleBubble becomes `<ModuleNode variant="bubble" intensity={…} accent={moduleColor} />` — the rgba literal goes away because the accent flows from MODULE_COLORS. Lock `COL_WIDTH`/`NODE_W` to one value across all three.

## 6. Multiple "ProgressRing" gauge implementations live alongside the shared `ProgressRing` primitive

- **Severity**: Medium
- **Category**: Component Architecture / Design System
- **File**: src/components/modules/evaluator/UnifiedSummaryView.tsx:345-404 (CombinedHealthGauge — bespoke arc) · ProjectHealthDashboard.tsx:54-68 (radar polar helpers, separate file) · GDDComplianceView.tsx:91 (ScoreRing — bespoke?) · HolisticHealthView.tsx:172,189,359 (uses shared ProgressRing) · PerformanceProfilingView.tsx (uses ProgressRing) · PatternLibraryView.tsx (uses ProgressRing)
- **Scenario**: `src/components/ui/ProgressRing.tsx` exists and is correctly used by HolisticHealth, Perf, Pattern, AssetScout. But UnifiedSummaryView ships a `CombinedHealthGauge` (a 270° arc with start/end angle math) that is *almost* a ProgressRing with `startAngle/sweepAngle` props missing. ProjectHealthDashboard ships a `polarToXY` helper for its radar chart — also independent. GDDComplianceView (line 91) references `<ScoreRing>` which is yet another local implementation. - **Impact**: The "combined project health" gauge on the Overview tab and the "overall completion" gauge in HolisticHealth render at adjacent moments in the same workflow; users perceive them as one metric in two places, but the shapes differ — full circle vs. 270° arc, different score-color thresholds (UnifiedSummary uses `healthColor` with 70/45/25 cutoffs; HolisticHealth uses 70/40 cutoffs). Migrate `CombinedHealthGauge` and `ScoreRing` to it. Centralize the score→color thresholds in `chart-colors.ts` as `function scoreToStatusColor(score: number)` with a single set of cutoffs (the AggregateQuality, ProjectHealth, UnifiedSummary, HolisticHealth files each have their own version of this exact function — five forks of one ramp).

## 10. AggregateQualityDashboard — quality-cell color ramp uses dark dirt colors that fail contrast on small text

- **Severity**: Medium
- **Category**: Polish / Accessibility
- **File**: src/components/modules/evaluator/AggregateQualityDashboard.tsx:30-43, 346-483
- **Scenario**: `qualityToColor()` returns hard-coded ramp colors `#7f1d1d` (red-900), `#78350f` (amber-900), `#14532d` (green-900) for the cell *background*. Inside the cell, `cell.label` (line 392) is rendered as `text-xs font-semibold text-text` (≈ `#eee`) — fine — but the status counts `text-[#4ade80]` / `text-[#fbbf24]` / `text-[#f87171]` (lines 473-475) sit on top of these dark backgrounds at `text-2xs`. The combination `#4ade80` (emerald-400) on `#14532d` (green-900) computes to ~3.1:1 contrast — below the WCAG AA 4.5:1 floor for small text. Same problem for `#fbbf24` on `#78350f` (~2.8:1). The legend swatches (lines 320-334) and the small star fills work, but the count digits — the actual data — do not.
- **Root cause**: The ramp was designed for the cell background; the foreground text colors were chosen against a neutral surface and never re-checked against the colored backgrounds.
- **Impact**: The "Module Quality Heatmap" is the centerpiece of the Quality dashboard. On the highest-density data (status counts in the corner of each cell), low-vision users and outdoor-light users cannot read the numbers. Sighted users notice the wash but parse it as "muted = not important," missing real signals.
- **Fix sketch**: Either (a) lighten the cell backgrounds — drop saturation by 30% so the foreground text retains contrast (`#7f1d1d` → `#3a1212` or use a token like `STATUS_ERROR + OPACITY_30` over `var(--background)`), or (b) keep the dark background and switch the count colors to `text-text` with colored *icons* only, or (c) wrap each count in a `bg-black/40 px-1 rounded` pill so the text gets a guaranteed-contrast micro-surface. Option (a) plus updated tokens is the cleanest.

## 11. CalendarRoadmap — variance chip and deadline state are visually similar to milestone progress; "overdue" not loud enough

- **Severity**: Low
- **Category**: Polish / Visual Hierarchy
- **File**: src/components/modules/evaluator/CalendarRoadmapView.tsx:377-388, 543-588
- **Scenario**: When a milestone is overdue (line 543, `isOverdue = variance < 0`), the SVG diamond marker switches from `STATUS_INFO` (blue) to `MODULE_COLORS.content` (a content-module pink, not a status red). The dashed line follows. The variance text label at line 580 uses the same pink. So "overdue" is rendered in a content-category accent rather than `STATUS_ERROR` / `STATUS_BLOCKER`. The variance chip on the milestone card (line 379-387) does the same thing in className-form: `text-amber-400` for overdue, `text-green-400` for buffer — neither matches the SVG's pink. So the same overdue milestone is amber on the card and pink on the Gantt.
- **Root cause**: `MODULE_COLORS.content` was reached for because it happens to be a red-ish tone, but it's a category color not a status color — and `STATUS_ERROR` was right there in the same import.
- **Impact**: Overdue milestones — the most actionable information on the roadmap — are visually muted (pink reads softer than red) and inconsistent between the card grid and the Gantt below it. A sighted user must learn two color codes for one concept.
- **Fix sketch**: Replace `MODULE_COLORS.content` at lines 555, 563-564, 580 with `STATUS_ERROR`. Replace `text-amber-400` at line 380 with `text-[color:var(--status-error)]` (or use the `STATUS_ERROR` token via inline style, like the rest of the file). Bonus: at significant overdue (e.g. variance < -7d), pulse the diamond with a subtle `animate-pulse` or wrap it in a `motion.g` with a soft glow — overdue is the rare, important signal and should out-shout the predicted bar.
