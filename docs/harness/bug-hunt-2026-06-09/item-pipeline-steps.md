# Bug Hunt — Item Pipeline Steps
> Total: 4
> Severity: 1 critical, 1 high, 2 medium, 0 low

## 1. Stale-closure double-produce drops a whole generation batch
- **Severity**: critical
- **Category**: race-condition
- **File**: src/components/layout-lab/steps/ItemArt.tsx:27-34 (and src/components/layout-lab/steps/shared/CliProduce.tsx:52-64)
- **Scenario**: If the artist double-clicks "Generate" (or clicks "Produce fix" then immediately re-rolls) on the Icon / 3D / Material steps before React re-renders, two handlers fire against the same render's closure...
- **Root cause**: `useGenerativeStep.generate()` closes over `history = readHistory(art?.data)`, captured once per render, and derives the new batch id from `history.batches.length` (`seq`) and builds the next state with `appendBatch(history, batch)`. Two clicks in the same render frame both read `batches.length === N`, both mint batch `bN` (and ids `bN-c0…`), and the second `produce()` call writes a `byEntity[...][step]` artifact built from the *same* stale `history` — so it overwrites the first batch's append instead of stacking on it. The synchronous CliProduce path (no `minDispatchMs` is passed by any Item step) never sets `dispatching`, so the `if (dispatching) return` guard at CliProduce.tsx:53 is dead — there is zero double-dispatch protection on the path every step actually uses. The whole premise of this module ("every batch is kept, never discarding prior re-rolls") is violated by its own dispatch path.
- **Impact**: data loss — a generated batch and its candidates (plus the recoverable art direction/prompt stamped on it) vanish silently; the gallery that promises "kept across re-rolls" loses history. Duplicate batch ids (`bN`) can also collide candidate ids, so a later `selectCandidate` resolves the wrong thumbnail.
- **Fix sketch**: Make `produce` updaters read prior state functionally instead of from a render closure — e.g. have the store expose an `appendBatch(entityId, step, batchFactory)` that computes `seq`/append inside `set((s) => …)` from live state, so concurrent dispatches serialize. Failing that, gate every CliProduce dispatch behind the in-flight `dispatching` lock unconditionally (not only when `minDispatchMs` is set), and derive batch ids from a monotonic counter or uuid rather than `batches.length`.

## 2. Divide-by-zero / NaN poisons the economy acceptance gate on hydrated data
- **Severity**: high
- **Category**: edge-case
- **File**: src/components/layout-lab/steps/itemsSteps.ts:57-79,144-166,309-315
- **Scenario**: If a persisted or server-hydrated Economy artifact has `power: 0` (or a `target`/`cost` that is missing/0 — e.g. a tuning reset, a partial AI write, or a legacy artifact) while the Acceptance gate and the Economy view render it...
- **Root cause**: `priceRatio(cost, power)` divides by `expectedPrice(power) = Math.round(power * 1.4)`, which is `0` for any `0 <= power < 0.36`, and `0/0` when cost is also 0 → `Infinity`/`NaN`. `economyCopy` then computes `Math.round(((power - target) / target) * 100)`; when `target` is missing, `Number(data.target)` is `NaN`, poisoning the whole expression. The only guard anywhere is `data.power == null` (accept:310, economyCopy:145) — it checks null/undefined but never `0`, never validates `target`/`cost`, and `Number(undefined)` silently becomes `NaN` rather than being rejected. Trust is placed in produced shape, but hydration (`hydrateEntity`) and the write-through sink can introduce artifacts this code never produced.
- **Impact**: corruption/UX degradation — the acceptance banner and Economy view render "price/power NaN×", "Power is ~NaN% above", and `ratio.toFixed(2)` = "Infinity"; `priceInBand`/`powerInBand` comparisons against `NaN` always return false, so an item is permanently and inexplicably flagged OUTLIER with nonsense numbers and the Produce-fix suggestion direction is gibberish.
- **Fix sketch**: Validate at the trust boundary: in `accept()` treat non-finite or `<= 0` `power`/`target` as the "not tuned" pending state (the same branch as `power == null`), and make `priceRatio` return a sentinel (or guard `expectedPrice` to a floor of 1) so the formula can never emit `Infinity`/`NaN`. Centralize a `Number.isFinite` check in the curve helpers so every call site is covered by construction.

## 3. Empty or colliding asset slug yields malformed / overwriting UE paths
- **Severity**: medium
- **Category**: data-loss
- **File**: src/components/layout-lab/steps/stepProps.ts:6-25 (consumed by itemsSteps.ts:383-388 and ItemArt.tsx:40,74,118)
- **Scenario**: If an item entity is named with only punctuation/non-ASCII (e.g. "—", an emoji, or a CJK name) or two items differ only in punctuation/spacing ("Iron Sword" vs "Iron-Sword" vs "IronSword")...
- **Root cause**: `slug(name)` is `name.replace(/[^a-z0-9]+/gi, '')` — it strips everything non-alphanumeric with no fallback and no uniqueness guarantee. An all-punctuation name collapses to `''`, producing `/Game/Items//T__Icon`; two differently-punctuated names collapse to the *same* slug. Every asset path (icon, mesh, material, montage, VFX, and the `DT_Items` row in `UE Packaging`) is keyed solely on this slug, and entity names can originate from user-staged drafts, not just well-formed seeds. The asset-path layout assumes names are ASCII-distinct after stripping — an assumption nothing enforces.
- **Impact**: data loss / corruption — two catalog items write to identical `/Game/Items/<slug>/` asset paths and the packaging manifest, so one item's cooked assets silently clobber another's; empty-slug items produce invalid UE paths (`//`, `T__Icon`) that fail or land in the wrong folder.
- **Fix sketch**: Make collisions impossible: incorporate the stable `entity.id` into the folder/asset key (`/Game/Items/<slug>_<shortId>/`), and have `slug()` fall back to a deterministic id-derived token when the alphanumeric result is empty. Reject or transliterate non-ASCII at the draft-creation boundary so the slug invariant holds before any path is computed.

## 4. "Report the reason it failed" is structurally unenforced — every Produce is success theater
- **Severity**: medium
- **Category**: silent-failure
- **File**: src/components/layout-lab/steps/shared/CliProduce.tsx:59-64 (and labPipelineStore.ts:54-63; all Item step `onComplete`/`runProduce`)
- **Scenario**: If the underlying CLI/asset generation a Produce dispatch represents does nothing, errors, or writes nothing meaningful, while the user clicks "Generate with CLI"...
- **Root cause**: Rule 4 ("production reports a result, or the reason it failed") is wired in the store (`fail()`, `LabStepArtifact.error`) and CliProduce (`validate`, `minDispatchMs`, the try/catch async path) — but no Item step uses any of it. None pass `validate` or `minDispatchMs`, none call `fail()`, and none render `art.error`. Every step takes the synchronous branch (CliProduce.tsx:59-64) which unconditionally sets `result = { ok: true, msg: note }` immediately after a fire-and-forget `onComplete(ctx)`, regardless of whether the produce was real. The acceptance gate is derived purely from the locally-written default payload, so it always flips to pass too. The contract exists only as dead code in this scope.
- **Impact**: UX degradation / silent failure — the UI shows a green "✓ written to the UE project + DB" and a passing acceptance gate for an operation that, in the real (non-prototype) wiring this models, may have failed or no-opped; there is no path by which a failure ever reaches the user. Errors are swallowed by design.
- **Fix sketch**: Route Item Produce through the async CliProduce path (`minDispatchMs`) with an `onComplete` that actually awaits the sink and throws on failure, and surface `art.error` in `StepFrame` when present. Better, make success non-optional to fake: have `produce`/the write-through sink return a result the step must reflect, so a step that does not report a real outcome cannot compile to a green banner.
