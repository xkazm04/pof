# Bug Hunt — AI Testing & Localization
> Total: 4
> Severity: 1 critical, 2 high, 1 medium

## 1. ON DELETE CASCADE silently no-ops — orphan scenarios leak forever because foreign_keys is never enabled
- **Severity**: critical
- **Category**: state-corruption
- **File**: src/lib/ai-testing-db.ts:36 (schema), :177-181 (deleteSuite), :185-205 (createScenario); src/lib/db.ts:11-19 (getDb)
- **Scenario**: A user creates a suite with N scenarios, then deletes the suite (`deleteSuite` → `DELETE FROM ai_test_suites WHERE id = ?`). The `ai_test_scenarios.suite_id ... ON DELETE CASCADE` clause is supposed to remove the children — but SQLite enforces foreign keys **per connection** and defaults them OFF. `getDb()` (the shared singleton connection) never runs `db.pragma('foreign_keys = ON')`, and `ensureAITestingTables()` doesn't either. So the cascade never fires: every deleted suite leaves its scenario rows behind permanently. The same gap lets `createScenario({ suiteId: <nonexistent> })` insert an orphan with no error (the `REFERENCES` clause is decorative), and `createScenario`'s `null` return (treated as "Suite not found" 404 in the route) can never trigger.
- **Root cause**: The schema relies on declarative referential integrity (`REFERENCES ... ON DELETE CASCADE`) while the runtime connection has FK enforcement disabled. Worse, two *unrelated* modules (`audio-asset-db.ts:62`, `asset-library-db.ts:69`) DO call `pragma('foreign_keys = ON')` on the *same shared connection*, so whether cascade works at all becomes non-deterministic — it depends on whether one of those `ensure*` functions happened to run earlier in the process lifetime. ai-testing's correctness must not hinge on another feature's init order.
- **Impact**: corruption / resource leak — unbounded orphan-row growth in `ai_test_scenarios`, dangling FKs, and behavior that flips between "cascade works" and "cascade leaks" across processes depending on unrelated module load order. Hard-to-reproduce data drift.
- **Fix sketch**: Enable `db.pragma('foreign_keys = ON')` once inside `getDb()` itself (so every table that declares an FK is enforced uniformly), rather than ad-hoc in scattered `ensure*` helpers. Then add an integration assertion that `deleteSuite` removes child scenarios. This makes "declared FK silently ignored" impossible across the whole DB layer.

## 2. Placeholder fallback "[DE] text" defeats the `untranslated` QA gate — fake translations pass QA and a locale shows "ready to ship"
- **Severity**: high
- **Category**: silent-failure
- **File**: src/lib/localization/translation-engine.ts:92-97 (fallback); src/lib/localization/qa-engine.ts:175-189 (untranslated check), :258-264 (readyToShip)
- **Scenario**: A string isn't in the canned `TRANSLATION_DB` and isn't a glossary term (the common case for any real corpus). `translateString` emits `translatedText = `[${locale.toUpperCase()}] ${source.sourceText}`` — e.g. `"[FR] Confirm"` — which is the untranslated English with a locale tag, not a translation. The `untranslated` QA check compares `tgt.trim().toLowerCase() === src.trim().toLowerCase()`; `"[fr] confirm" !== "confirm"`, so the check **passes**. The placeholder also preserves all source numbers/`{0}` tokens, so `number_parity` and `placeholder_parity` pass too. Net: the entry produces **zero blocking QA findings**.
- **Root cause**: The QA layer was built to catch "translation identical to source = untranslated," but the engine's own fallback disguises the untranslated string with a prefix the QA doesn't normalize away. The detector and the thing it's meant to detect were designed independently. Because `qaBlocksShip` only blocks on critical/warning findings and the fallback yields none, `LocaleQAStatus.readyToShip = entries.length > 0 && blockingCount === 0` can go **green** for a locale composed entirely of `[XX] …` placeholders.
- **Impact**: corruption / shipping risk — a localized build can be greenlit "ready to ship" while every string is actually English with a `[FR]`/`[JA]` prefix baked into the game UI. The single most dangerous outcome the QA pass exists to prevent, silently bypassed.
- **Fix sketch**: Mark fallback output as a first-class state instead of smuggling it through `translatedText`: set status `pending`/`machine_stub` and have QA treat a recognizable stub prefix (or a `synthetic: true` flag on the entry) as a blocking `untranslated` finding. QA should strip/ignore engine-internal tagging before comparison so "looks different but isn't a real translation" can never read as clean.

## 3. Per-keystroke whole-array PUT + non-optimistic refetch → stale-closure lost updates and input reverting while typing
- **Severity**: high
- **Category**: race-condition
- **File**: src/components/modules/game-systems/AITestingSandbox.tsx:263-266 & 282-285 (handlers), :406-410 (onChange); src/hooks/useCRUD.ts:75-84 (mutate); src/components/modules/game-systems/AIBehaviorView.tsx:89-94 (handleUpdateScenario)
- **Scenario**: Typing in a stimulus/expected-action field fires `onChange` → `handleUpdateStimulus(idx, { description })` → rebuilds the **entire** `scenario.stimuli` array from the current props snapshot → `onUpdateScenario(id, { stimuli })` → `updateScenario` → `mutate` PUT. `useCRUD.mutate` has **no optimistic update**: it `await apiFetch(PUT)` then `await refetch()` (full GET) and replaces `data.suites` wholesale. Two quick keystrokes race: keystroke B reads `scenario.stimuli` from props that haven't yet been refreshed by A's refetch, so B's PUT overwrites A's change (lost update). When a refetch resolves mid-typing, the controlled `value={stim.description}` snaps back to server state, dropping characters typed after the in-flight request was sent.
- **Root cause**: A coarse-grained "replace whole row on every keystroke, then re-pull the whole collection" mutation model applied to a high-frequency text input. Each edit serializes a stale full array, and the refetch-replaces-state pattern fights the user's cursor. There's no debounce, no field-level patch, and no optimistic merge.
- **Impact**: data loss + UX degradation — silently dropped edits to stimuli/expected-action text and visible character loss / cursor jumps while typing.
- **Fix sketch**: Make this class impossible by decoupling local edit state from server sync: keep edits in component state, debounce (or commit on blur) a single PUT, and either send a field-level patch or apply an optimistic update so refetch can't clobber in-flight typing. A functional updater for the array (not a props-snapshot read) closes the stale-closure gap.

## 4. Translation auto-apply status is non-deterministic w.r.t. locale/string ordering because one RNG stream is shared across the whole batch
- **Severity**: medium
- **Category**: logic-error
- **File**: src/lib/localization/translation-engine.ts:129 (single rng), :143-146 (nested loop), :88-96 (confidence), :109 (status gate)
- **Scenario**: `translateBatch` creates one `mulberry32(seed)` stream and consumes it inside `for (str) { for (locale) { translateString(...rng...) } }`. Each non-glossary translation's confidence is `0.6 + rng()*0.2` (fallback) or `0.85 + rng()*0.1` (DB hit), and `status = confidence >= autoApplyThreshold ? 'translated' : 'needs_review'`. Because every `rng()` advance is shared and order-dependent, the confidence assigned to a given (string, locale) pair depends on how many draws preceded it — i.e. on the **order of `config.targetLocales`** and the string list. Reorder the locales in config (a user-editable array) and the same translation can get a different confidence, flipping it across `autoApplyThreshold` — so it auto-applies in one run and is held for review in another, with no input change.
- **Root cause**: A "seeded for reproducibility" design that ties reproducibility to iteration order rather than to the (string, locale) identity. The per-pair confidence should be a pure function of its own inputs, not of its position in a shared sequence.
- **Impact**: UX degradation / trust — the review queue and per-locale ship decisions become unstable under harmless config reordering; "deterministic" results aren't. Bites hardest whenever `autoApplyThreshold` is lowered from the 0.85 default (which the config + API fully support), where fallback confidences straddle the gate.
- **Fix sketch**: Seed per entry — `mulberry32(hashString(`${stringId}:${locale}:${seed}`))` — so each (string, locale) confidence is independent of loop order. This makes ordering-induced status flips structurally impossible while preserving reproducibility.
