# Bug Hunt — Progression & Save Systems
> Total: 4
> Severity: 0 critical, 2 high, 2 medium, 0 low

## 1. Data Recovery flow renders as "already complete" on mount (success theater)
- **Severity**: high
- **Category**: logic-error
- **File**: src/components/modules/core-engine/sub_save/advanced/DataRecoveryTool.tsx:15
- **Scenario**: An operator opens the Advanced → Data Recovery tool to recover a corrupted save. Before they click anything — before any detect/recover/verify step has run — every step indicator already shows the green "complete" check, the wizard reads as finished, and the confidence gauge + per-field results ("recovered / partial / lost") are fully populated.
- **Root cause**: The wizard step is initialized to the *terminal* step: `useState<RecoveryStep>('confirm')`. The step list order is `detect → recover → verify → confirm`, and the indicator computes `stepIndex = RECOVERY_STEPS.findIndex(s => s.id === recoveryStep)` then marks `isComplete = i <= stepIndex` (line 34-36). With the current step pinned to the last index, *all* prior steps satisfy `i <= stepIndex` and render complete from the very first paint. The design assumption — that the initial state represents "nothing done yet" — is inverted: the start state is the end state. There is also no effect/handler that actually advances or runs a recovery; clicking a step only `setRecoveryStep(step.id)`, so an earlier step can never truthfully "un-complete" the flow either.
- **Impact**: UX degradation bordering on data-integrity misrepresentation — a save-recovery surface tells the user recovery succeeded (incl. fabricated per-field confidence) when nothing ran. On a tool whose entire job is to communicate trustworthy recovery status, "success theater" can lead an operator to accept a partially/never-recovered save.
- **Fix sketch**: Initialize to the first step (`'detect'`) and model "completed" as an explicit, monotonically-advanced index that only increases when a real step finishes; make the whole flow a state machine where the result panels render only once `verify`/`confirm` is actually reached. The class of bug (start-state == end-state) is eliminated by never deriving "complete" from a hardcoded terminal default.

## 2. XP curve never reaches MAX_LEVEL — chart and "Max XP" silently understate the curve
- **Severity**: medium
- **Category**: edge-case
- **File**: src/components/modules/core-engine/sub_progression/_shared/data.ts:34
- **Scenario**: A designer drags the Base XP / Exponent sliders and reads the header "Max Level: 50 | Max XP: …" plus the plotted endgame curve to tune late-game pacing. The number shown and the curve's right edge are the values for **level 46**, not 50.
- **Root cause**: `generateChartData` samples with `for (let lvl = 1; lvl <= MAX_LEVEL; lvl += 5)`, producing levels `[1, 6, 11, …, 46]` — the step-of-5 starting at 1 lands on 46 and stops, so MAX_LEVEL (50) is never sampled. The dead-code clamp `const levelToUse = lvl > MAX_LEVEL ? MAX_LEVEL : lvl` proves the author intended the last sample to be 50, but the loop bounds make that branch unreachable. Downstream, `MainChartArea` derives `maxXp = chartData[last].xp` (level 46) and labels it as the curve's Max XP, while `CurveDeltaSummary` independently computes "Time to Max" via `calculateXpForLevel(MAX_LEVEL=50, …)` — so two surfaces of the same tuning panel disagree on what "max" means.
- **Impact**: Corrupted balance data on a balance-authoring tool — the displayed Max XP is wrong (it's the level-46 value), the curve omits its final 4 levels, and the compare-mode delta summary is computed against a different endpoint than the chart, making cross-panel comparisons inconsistent.
- **Fix sketch**: Sample levels from a single source of truth that is guaranteed to include the endpoints (e.g. `[1, …step…, MAX_LEVEL]` with the final value forced to MAX_LEVEL, or iterate `lvl = 0; lvl <= MAX_LEVEL` / push MAX_LEVEL after the loop). Derive every "max" label and the delta endpoint from that same array so the chart, header, and summary can't diverge.

## 3. Power-curve danger zones assume two independent series are equal-length & index-aligned
- **Severity**: medium
- **Category**: edge-case
- **File**: src/components/modules/core-engine/sub_progression/analysis/PowerCurveDangerZones.tsx:70
- **Scenario**: A caller renders `<PowerCurveDangerZones playerPower={…} enemyDifficulty={…} />` (both are public props) with series of different lengths — e.g. 11 player samples but 9 enemy samples after trimming a shorter enemy ramp.
- **Root cause**: The zone-coloring loop iterates over `playerPower` but cross-indexes the *other* series: `const ratio = safeDivide(playerPower[i], enemyDifficulty[i])` (line 74). The `plottable` guard only checks that *each* series has ≥2 points and some spread — it never asserts the two arrays are the same length or share an x-axis. When `enemyDifficulty` is shorter, `enemyDifficulty[i]` is `undefined`; `safeDivide(n, undefined)` yields `NaN/…` → collapses to `0`, so those segments are silently painted as the "Hard" (bad) zone regardless of the real balance. The two polylines are each plotted against their own index span, so a length mismatch also misaligns the lines horizontally without any warning.
- **Impact**: Silent data corruption of the visualization — danger zones (the whole point of the chart) are mis-colored and the two curves drift out of x-alignment, with no empty-state or error to signal it. The shipped constant defaults are equal-length so it never triggers in-app, but the trust boundary (props) is unvalidated.
- **Fix sketch**: Validate at the prop boundary — require `playerPower.length === enemyDifficulty.length` (fold the equality into `plottable`, falling back to the empty state otherwise), and zip the two series into a single `[{ level, player, enemy }]` array up front so every consumer indexes one aligned structure instead of two parallel arrays.

## 4. Auto-save trigger checkboxes and toggles are non-functional controls (dead config UI)
- **Severity**: high
- **Category**: silent-failure
- **File**: src/components/modules/core-engine/sub_save/advanced/AutoSaveConfig.tsx:46
- **Scenario**: A designer opens Auto-Save Config to disable "Boss Kill" saves (combat-save risk) and enable "Level Up" saves. They click the trigger rows' checkbox boxes and the Combat-save / Compression toggles. Nothing changes — and crucially, nothing tells them the click was ignored.
- **Root cause**: Only the interval slider is wired to state (`useState(AUTO_SAVE_CONFIG.intervalSeconds)`). The trigger list renders checkbox-styled boxes whose checked/unchecked appearance is driven purely by the static constant `trigger.enabled` with **no `onClick`/`onChange`** (lines 46-69), and the Combat-save / Compression "switches" are likewise rendered from `AUTO_SAVE_CONFIG.combatSaveEnabled` / `.compressionEnabled` constants with no handler. The controls *look* interactive (checkbox affordance, toggle pill with a sliding knob, `cursor` styling) but are render-only. The design assumption — that a control that looks toggleable persists a setting — is false; the state simply isn't lifted.
- **Impact**: Silent failure / UX degradation on a settings surface — users believe they've configured auto-save trigger policy and combat-save/compression behavior, but every change is silently discarded with no feedback, no persistence, and no disabled/read-only cue. On a save-system config panel this can lead to a false belief that, e.g., combat auto-save is enabled when it is not.
- **Fix sketch**: Either (a) lift trigger/toggle state into `useState`/store with real `onChange` handlers that persist (matching the interval slider), or (b) if this is intentionally a read-only status view, strip the interactive affordances (remove checkbox/toggle chrome, render as static badges) so no control implies an action it doesn't perform. Make "looks interactive ⇒ is interactive" an invariant enforced by using a shared Toggle/Checkbox component that *requires* an `onChange`.
