# Bug Hunt — Harness Autonomous Builder
> Total: 4
> Severity: 1 critical, 3 high, 0 medium, 0 low

## 1. Checkpoint `git reset --hard` runs while concurrent sibling sessions are mid-write — wipes their uncommitted work
- **Severity**: critical
- **Category**: data-loss
- **File**: src/lib/harness/orchestrator.ts:382-392, 642-662 (rollbackBeforePromote / runStreamingPool); checkpoint.ts:191-197
- **Scenario**: A user starts the harness with `--checkpoint` (or `checkpoint: true` via the API) and the default concurrency. `createDefaultConfig` defaults `maxConcurrent` to 1, but both real entry points override it: `run-harness.ts` sets `maxConcurrent: parseInt(args['concurrency'] ?? '4')` (line 198) and even the API `start` path leaves the default-config concurrency in force only when no executor is passed. With concurrency 4, area A exhausts its retries while areas B, C, D are still inside live `claude -p` sessions actively editing files. The pool calls `rollbackBeforePromote(A)` → `checkpointer.rollbackToLastGreen()` → `git reset --hard <lastGreenSha>` in the shared project tree.
- **Root cause**: Checkpoint rollback and the concurrency setting are two independent config knobs with no mutual-exclusion guard. The design assumes "checkpoint ⇒ sequential" (the code comment in checkpoint.ts:15-17 and types.ts:256 literally says "Assumes sequential execution (maxConcurrent = 1)"), but nothing enforces it: `createHarnessOrchestrator` never clamps `maxConcurrent` to 1 when `checkpoint === true`, and `runStreamingPool` issues the hard reset without checking whether other areas are active in the pool.
- **Impact**: corruption / data loss — `git reset --hard` discards every uncommitted edit in the working tree, including the half-written files of 1–3 concurrently running areas. Those sessions then continue writing on top of a reset tree, producing a corrupted mix of pre-reset and post-reset state. Hours of generated work and any user-side uncommitted changes are silently destroyed.
- **Fix sketch**: Make the unsafe combination impossible: when `config.checkpoint === true`, force `maxConcurrent = 1` inside `createHarnessOrchestrator` (and log it), OR gate every `git reset --hard` behind an assertion that `active.size === 0`. Better still, replace the destructive working-tree reset with a `git stash`/worktree-scoped rollback that only touches the failed area's paths, so a sibling's in-flight edits can never be in the blast radius.

## 2. Budget cap is silently bypassed whenever the CLI does not report a per-session cost
- **Severity**: high
- **Category**: silent-failure
- **File**: src/lib/harness/orchestrator.ts:369-375 (recordSessionCost), 88-98 (avgSessionCost / budgetWouldOverflow); claude-session.ts:137-138
- **Scenario**: A user sets `budgetUsd: 20` expecting the run to pause near $20. The harness spawns dozens of expensive sessions and never pauses, blowing far past the cap, because the cost the governor relies on is never recorded.
- **Root cause**: `recordSessionCost` early-returns when `costUsd` is falsy (`if (!costUsd || costUsd <= 0) return;`), so `sessions` and `spentUsd` stay at 0 unless every session reports a positive cost. The cost only gets populated if the stream-json emits `parsed.type === 'result'` with a `cost_usd` field (claude-session.ts:138). Current Claude Code CLIs report the field as `total_cost_usd`, not `cost_usd`, so `costUsd` is perpetually `undefined`. With `spentUsd = 0` and `sessions = 0`, `avgSessionCost` returns 0 and `budgetWouldOverflow` evaluates `0 + 0 > budgetUsd` → always false. The governor's `wouldOverflowNow()` therefore never fires. Compounding it, self-heal sessions (`attemptSelfHeal`) never call `recordSessionCost` at all, so their spend is invisible by construction even when the field name is correct.
- **Impact**: security / cost blow-out — the single safety rail meant to bound autonomous spend is a no-op under the most common CLI output shape, so an unattended run can incur unbounded API cost.
- **Fix sketch**: Treat a missing cost as a *known unit cost*, not zero: when the CLI reports no cost, increment `sessions` and add a configured per-session estimate so `avgSessionCost`/projection still advance toward the cap. Parse both `cost_usd` and `total_cost_usd` (and `usage`-derived cost) in claude-session.ts. Route self-heal spend through `recordSessionCost` too. Make the governor fail-closed: if budget is set but no cost signal has ever arrived after N sessions, pause and surface a warning rather than running forever.

## 3. Self-heal reports "healed" without verifying when no verify command exists (success theater)
- **Severity**: high
- **Category**: silent-failure
- **File**: src/lib/harness/orchestrator.ts:211-247 (attemptSelfHeal), 466-473 (re-verify branch)
- **Scenario**: A required gate fails (e.g. the visual gate, or a custom gate configured with no `command`). The orchestrator runs `pickHealVerifyCommand`, which returns `null` because the failing gate carries no command and no typecheck/build gate is present. `attemptSelfHeal` spawns one fix session, then — because `verifyCommand` is null — returns `true` unconditionally (line 238-242). The caller treats `healed === true` and re-runs `verify`, but the underlying failure was never actually re-checked by the heal step itself; the area can be reported as self-heal "SUCCEEDED" purely on optimism.
- **Root cause**: The "optimistically resolve as healed" branch conflates *"we could not verify"* with *"the fix worked."* `healed` is overloaded to mean both "verification passed" and "we gave up checking," so a non-result is laundered into a positive signal. The same overload exists in the `exec(verifyCommand, …)` path, which resolves `err === null` — any non-zero exit, timeout, or spawn error all collapse to "not healed," but a command that exits 0 for unrelated reasons (e.g. `ls Source/` on a tree where the errors are elsewhere) reports a false heal.
- **Impact**: corruption of run state / UX degradation — areas advance and emit green `harness:learning` / `harness:area-completed` events on the strength of fixes that were never confirmed, so the final guide and pass-rate claim work that may still be broken. A user trusts a "self-heal SUCCEEDED" log that means nothing.
- **Fix sketch**: Return a tri-state from `attemptSelfHeal` (`healed | unverified | failed`) instead of a boolean. Only the re-run of the *actual failing gate* — not a proxy command and not "no command" — may flip an area to healed. When verification is impossible, keep the area in its failed/retry state and emit an explicit "could not verify heal" event rather than advancing.

## 4. Feature-name mismatch force-marks every feature in an area as passing
- **Severity**: high
- **Category**: state-corruption
- **File**: src/lib/harness/orchestrator.ts:481-509 (feature reconciliation)
- **Scenario**: A session emits a `@@HARNESS_RESULT` JSON whose `features[]` all say `"status":"pass"` but whose `name` values don't match any planned feature (the model renamed them, themed them via `themeDirective`, or summarized them differently). The fuzzy matcher at lines 485-491 finds nothing, so `matchedFeatures` stays 0. The fallback at line 501-508 then fires: `allParsedPass && matchedFeatures === 0` → it loops over `area.features` and force-sets **every** feature to `status:'pass', quality:4`, regardless of what was actually built.
- **Root cause**: The fallback assumes "the model said everything passed, so even if I can't map names, trust it globally." It uses the model's self-reported aggregate as ground truth for features it could not individually correlate. Combined with the area-success rule (`featurePassRate >= areaPassThreshold`), this hands a passing pass-rate to an area whose features were never verified by name — and `updatePlanStats` then counts them toward the global `targetPassRate` stop condition.
- **Impact**: data corruption of the plan + premature completion — the harness can declare the whole game "90% passing" and stop, or generate a guide claiming features are done, on the basis of an unmatched, unvalidated model summary. The verification gates only check build/lint/visual, not per-feature truth, so nothing else catches it.
- **Impact knock-on**: because failed-but-promoted areas also get `quality:3` defaults (lines 522-528), real signal about what actually works is lost.
- **Fix sketch**: Remove the "match nothing ⇒ pass everything" shortcut. If `matchedFeatures === 0` while `parsed.features.length > 0`, treat it as a *parse/correlation failure* (the names don't describe this area) — record a partial/failed outcome and keep features `pending`, never silently green. Optionally require the model to echo each planned feature's exact `id` (already present on `PlannedFeature`) so matching is exact rather than fuzzy substring.
