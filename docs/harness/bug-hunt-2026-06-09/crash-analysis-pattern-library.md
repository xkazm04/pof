# Bug Hunt — Crash Analysis & Pattern Library
> Total: 4
> Severity: 1 critical, 2 high, 1 medium

## 1. Imported crashes never enter pattern detection, stats, or the cached analysis
- **Severity**: high
- **Category**: state-corruption
- **File**: src/lib/crash-analyzer/analysis-engine.ts:187 (cache) + src/app/api/crash-analyzer/route.ts:24 (parse-log) + src/stores/crashAnalyzerStore.ts:76 (importCrashLog)
- **Scenario**: A user pastes a real UE5 crash log in the Import tab. `POST {action:'parse-log'}` calls `parseCrashLog` + `analyzeSingleCrash`, returns one report, and the store appends it to `reports` client-side. But the imported crash is never persisted anywhere — there is no DB write and `analyzeAllCrashes()` only ever reads the static `SAMPLE_CRASHES`. `analyzeAllCrashes` also memoizes its result into module-level `cachedResult` on first call, so even if a persistence path existed, every subsequent GET / `analyze` / `full-analysis` returns the frozen sample-only snapshot for the process lifetime.
- **Root cause**: The design assumes "import = display one row", but the feature is presented as crash *analysis*: pattern detection (`detectPatterns` needs 2+ matching signatures), `CrashStats` (Total/Critical/Patterns/Systemic/Recent counters), and the Health Map all derive exclusively from the sample set. Imports are a client-only optimistic append with no server source of truth, and the permanent memoization means the analysis can never reflect them. Worse, the imported report's `culpritFrame`/`mappedModule` feed nothing, and `analyzeSingleCrash` only matches a diagnosis if `d.crashId === report.id`, but imported IDs are `crash-<Date.now()>` which can never be in `SAMPLE_DIAGNOSES` — so imports are always diagnosis-less.
- **Impact**: Success theater / silent data loss — users believe imported crashes are being analyzed and folded into patterns/stats; in reality they vanish on refresh, never affect counters, and never join a pattern. The whole Import workflow is non-functional past a single render.
- **Fix sketch**: Persist imported crashes to a SQLite table (mirroring `error_memory`/`pattern_library`), have `analyzeAllCrashes` union persisted + sample reports, and replace the permanent `cachedResult` with a cache that invalidates on any write (or drop it — pattern detection over a bounded set is cheap). Make `parse-log` insert, then return the recomputed analysis so the store reconciles instead of appending.

## 2. `parseCrashLog` misclassifies every access violation as a null-pointer deref and crashes on unparseable timestamps
- **Severity**: medium
- **Category**: logic-error
- **File**: src/lib/crash-analyzer/analysis-engine.ts:240
- **Scenario**: A user imports a genuine `EXCEPTION_ACCESS_VIOLATION reading address 0x000000ABCD1234` (a wild non-null pointer). The first branch `if (errorMessage.includes('ACCESS_VIOLATION') || errorMessage.includes('address 0x000000000000'))` fires on the `ACCESS_VIOLATION` substring alone, so `crashType = 'nullptr_deref'` and `severity = 'critical'`. The very next `else if (errorMessage.includes('ACCESS_VIOLATION'))` for the real `access_violation` type is therefore dead code — it can never be reached.
- **Root cause**: The two conditions are ordered most-general-first instead of most-specific-first. The intent was clearly "null deref = access violation AT a near-zero address; otherwise generic access violation", but the `||` collapses both into the first branch. Every access violation is now reported as a null-pointer dereference, which also drives a wrong `critical` severity and a wrong plain-English "what happened/what to do" story in the UI.
- **Impact**: UX degradation / misdiagnosis — operators are told a heap-corruption or use-after-free access violation is a null deref, steering the one-click fix prompt and triage toward the wrong root cause class.
- **Fix sketch**: Reorder to test the specific near-null address signature first (`includes('address 0x0000000000000000')` → nullptr_deref), then fall through to `access_violation` for any remaining `ACCESS_VIOLATION`. Generally, classify by most-specific signature before generic family, and add a unit test asserting a non-null-address access violation maps to `access_violation`.

## 3. Pattern/anti-pattern extraction silently double-counts and pollutes results when sessions span multiple projects
- **Severity**: high
- **Category**: logic-error
- **File**: src/lib/pattern-extractor.ts:183 + src/lib/pattern-extractor.ts:391
- **Scenario**: `extractPatterns` computes `successRate = successful.length / sessions.length` once per module and stamps it onto *every* approach cluster in that module, while `sessionCount` is set to the per-cluster size. So a module with one 100%-success approach and one 0%-success approach reports BOTH patterns with the blended module-wide success rate, and `projectCount` is hardcoded to `1` (line 216) regardless of how many projects the sessions came from. `confidence` (`computeConfidence`) and the suggestion ranking (`suggestPatterns` weights `successRate*40`, `projectCount*5`) are computed from these wrong numbers.
- **Root cause**: The success rate is a module-level aggregate applied at cluster granularity — the denominator is "all sessions in the module" not "sessions using this approach". A failing approach inherits the module's overall success rate, so a dangerous approach can present as "70% success, promising" and be recommended. (Contrast: the anti-pattern path at line 388-391 correctly recomputes `failureRate` per-approach — proving the per-cluster rate is what was intended.) `projectCount=1` also defeats the "Proven across N projects" diversity bonus that the suggestion engine and confidence ladder rely on.
- **Impact**: Corruption of the recommendation surface — mined patterns carry inflated/blended success rates and confidence, so `suggestPatterns` can surface a low-quality approach as canonical-adjacent, actively steering authors toward worse implementations. The "promising"/"proven" badges become untrustworthy.
- **Fix sketch**: Compute `successRate` per approach cluster: `clusterSuccesses / sessionsUsingThatApproach` (classify every session's approach once like the anti-pattern path already does), and derive `projectCount` from a distinct project_id over the cluster rather than hardcoding `1`. Make confidence/ranking consume only per-cluster metrics so a failing approach can never inherit a sibling's success rate.

## 4. Anti-pattern trigger keywords match unanchored substrings, firing false warnings on unrelated prompts
- **Severity**: critical
- **Category**: logic-error
- **File**: src/lib/pattern-library-db.ts:639 + src/lib/pattern-extractor.ts:331
- **Scenario**: `extractTriggerKeywords` harvests any 4+ char word appearing in >50% of failed prompts (e.g. `state`, `event`, `component`, `actor`, or even a stray token like `cast`). `checkPromptForAntiPatterns` then matches with `lower.includes(kw.toLowerCase())` — an unanchored substring test. A future prompt "Add a **state**ful HUD widget" matches the keyword `state`; "broad**cast** the score" matches `cast`. With a small `triggerKeywords` array the per-keyword match drives `matchScore` over the 30% gate, so an unrelated prompt gets a scary blocking-style warning: "This approach failed 85% of the time… switch to it?".
- **Root cause**: Two compounding assumption failures: (a) keyword *generation* keeps generic high-frequency English/UE tokens (the stopword filter only excludes a dozen words), and (b) keyword *matching* uses substring `includes` with no word-boundary anchoring, so short keywords match inside larger words. The match-score denominator (`matchedKeywords / triggerKeywords.length`) means a single noisy keyword in a short trigger list yields a high confidence. There is no minimum-distinctiveness or boundary guard at the trust boundary between user prompt and the anti-pattern catalog.
- **Impact**: UX degradation eroding trust in the whole guardrail — users are repeatedly warned away from correct approaches by anti-patterns mined from unrelated failures, training them to ignore all warnings (so a true anti-pattern is also ignored). Because the warning copy implies the *current* approach failed 85%, it can cause a user to abandon a sound plan.
- **Fix sketch**: Anchor matching to whole words (regex `\b<kw>\b` or tokenize both sides into word sets and intersect), require keywords to be domain-distinctive (drop a curated stopword/common-token list, require min length and a TF-IDF-style rarity threshold across all sessions), and require ≥2 distinct keyword hits (or a higher match-score floor) before surfacing a warning. This makes "generic word collision → false anti-pattern warning" structurally impossible.
