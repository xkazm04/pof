# Bug Hunt — Bestiary & Enemy Design
> Total: 4
> Severity: 0 critical, 2 high, 2 medium, 0 low

## 1. Elite-modifier stat math clamps to [0,100], silently swallowing buffs and corrupting the diff readout
- **Severity**: high
- **Category**: logic-error
- **File**: src/components/modules/core-engine/sub_bestiary/_shared/data.ts:347-361 (consumed at src/components/modules/core-engine/sub_bestiary/archetypes/ArchetypeBuilder.tsx:140-159)
- **Scenario**: A designer opens the Archetype Builder / card-modifier UI and stacks elite modifiers on a high-stat enemy. They apply "Commander" (+60% HP) to a stat already at 100, or "Shielded" (+30 flat Shield HP) to any enemy whose HP slider is at/near 100. The preview shows the buff as `+0` (no change), and the effective bar does not move.
- **Root cause**: `applyModifiers` ends with `Math.max(0, Math.min(100, value))`. The 0–100 clamp is treated as a UI normalization, but the stats it operates on are *also* the authoring values that drive the preview diff (`diff = effective - value` in `BuilderPreview`). Multiplicative/additive elite mods routinely push values past 100 (100 × 1.6 = 160, 95 × 1.25 = 118, 100 + 30 = 130), so any buff that crosses the ceiling is clamped away and reported as no-op. Worse, because the clamp is applied to the *final* accumulated value, a stack of "+50% then −20%" yields a different result than "−20% then +50%" only matters below the ceiling — but at the ceiling the order/visibility breaks entirely, so the designer cannot tell a strong modifier from a useless one. The codegen path (`generateModifierGE`) emits the *unclamped* real multipliers, so the C++ a designer ships diverges from the preview they balanced against.
- **Impact**: UX degradation + balance corruption — designers tune elite tiers against a preview that lies about high-end stats; the strongest legendary modifiers appear identical to weak ones, and the generated GE does not match what was previewed.
- **Fix sketch**: Separate the *model* from the *display*. Compute and return the true effective value (unclamped) from `applyModifiers`; clamp only at the rendering layer (e.g. bar fill `pct = Math.min(100, effective)`), and show the real numeric `effective`/`diff`. Make the class of bug impossible by never letting the presentation clamp feed back into the value used for diffs/codegen.

## 2. Wave-list virtual scroller keeps a stale `scrollTop`, so filtering to a smaller tier can render a blank list
- **Severity**: high
- **Category**: state-corruption
- **File**: src/components/modules/core-engine/sub_bestiary/encounters/EncountersTab.tsx:35-49,123-150
- **Scenario**: The user scrolls the 24-wave Wave Choreographer to the bottom (`scrollTop` ≈ 800px), then picks a tier filter such as "raid-boss" (1 wave) from the dropdown. The visible window collapses but `scrollTop` state is still ~800.
- **Root cause**: The virtual window is computed from a React state copy of the scroll position: `startIdx = floor(scrollTop/36) - overscan`, `endIdx = ceil((scrollTop+containerH)/36) + overscan`, with `scrollTop` only updated by the container's `onScroll`. When `waveFilter` changes, `filteredWaves` shrinks and `containerH` recomputes, but `scrollTop` is never reset. The browser clamps the real DOM `scrollTop` to the new (smaller) scrollable height and does **not** always fire a synthetic `scroll` event, so the stale state persists. `filteredWaves.slice(startIdx, endIdx)` then slices past the end of the new array → `startIdx` can exceed `filteredWaves.length`, yielding an empty slice. The assumption "state scrollTop tracks the DOM scrollTop" is violated whenever the content height changes underneath it.
- **Impact**: UX degradation — the wave timeline appears empty (or shows only overscan rows positioned off-screen) after narrowing a filter; the user must manually scroll to "wake" the list, and may conclude the filtered tier has no waves.
- **Fix sketch**: Reset scroll on filter change — in the `setWaveFilter` handler set `scrollRef.current.scrollTop = 0` and `setScrollTop(0)` together (or key the scroll container by `waveFilter` to remount it). Generally, derive `startIdx`/`endIdx` from a clamped `Math.min(scrollTop, maxScroll)` so the window can never index past the current list.

## 3. AI-coverage linter keys off behavior-tree summary *keys*, producing false "missing"/"covered" verdicts
- **Severity**: medium
- **Category**: silent-failure
- **File**: src/lib/bestiary/ai-coverage.ts:23-34
- **Scenario**: Run `lintAiCoverage` on any KOTOR/expanded archetype whose `btSummary` uses the project's actual schema — keys `Sense` / `Attack` / `Special` (see `_shared/data.ts` and `data-expanded.ts`), e.g. `{ Sense: 'Force sense 12m', Attack: 'Force lightning then melee', Special: '...' }`. The linter reports aggro, patrol, and retreat as **not covered**, even though `Sense` is exactly the perception/aggro behavior.
- **Root cause**: The matcher builds `text = `${k} ${v}`.toLowerCase()` and tests `keywords.some(kw => text.includes(kw))`, i.e. it matches against the *key name* as well as the value. The CORE_BEHAVIORS keyword lists were written for the four-melee combat archetypes whose keys happen to be `Idle/Patrol/Chase/Attack`, but the bulk of the roster uses `Sense/Attack/Special`. None of those keys/values contain `aggro|aggress|detect|perceive|sight|chase` (`Sense` ≠ any keyword), `patrol|idle|wander|waypoint|guard`, or `retreat|flee|reposition|kite|escape`, so ~80 archetypes are reported as missing three of four core behaviors. Symmetrically, matching on the key produces false positives: an entry `{ Patrol: 'sprint in and attack' }` is counted as patrol coverage purely because the *key* says "Patrol", regardless of what the AI actually does.
- **Impact**: Silent wrong analysis — the coverage facet would tell designers their fully-specified enemies have no detection/patrol/retreat logic (or vice-versa), driving wasted "fixes." (The function is currently pure/unwired, so no live crash, but it is the documented driver of the BestiaryAiFacet coverage view and is wrong on contact.)
- **Fix sketch**: Match on the *value* only (`v.toLowerCase()`), and broaden the keyword sets to the real schema vocabulary (add `sense`, `proximity`, `visual`, `sonar`, `scent`, `vibration`, `echolocation` to aggro; `special`, `regenerat`, `flee` to retreat; etc.). Better: have archetypes declare structured behavior tags instead of free-text summaries so coverage is a set-membership check, not keyword guessing.

## 4. Archetype grid uses filtered-list length as a change proxy, so equal-count filters leave the user on an out-of-range (blank) page
- **Severity**: medium
- **Category**: edge-case
- **File**: src/components/modules/core-engine/sub_bestiary/archetypes/ArchetypesTab.tsx:55-60,92-99
- **Scenario**: With ~96 archetypes and `PAGE_SIZE = 24`, the user pages to page 3 (items 73–96), then changes a filter to a *different* selection that also yields the same number of results as the current filtered set (e.g. switching between two areas/categories that each return the same count). The grid shows nothing.
- **Root cause**: Page reset is gated on `prevFilterLen !== filteredArchetypes.length` — it uses the result *count* as a stand-in for "the filter changed." Two distinct filter states with identical cardinality compare equal, so `setPage(0)` never fires. `paginatedGroups` then slices `filteredArchetypes.slice(page*24, page*24+24)`; if the new (different but equal-length) result set is shorter than `page*24`, or simply doesn't have a page 3 conceptually, the slice is empty and the grid renders the empty state despite matching items existing on page 1. Even the same-length-but-reordered case strands the user on a later page they didn't intend. The design assumption "length identity ⇒ no change" is false.
- **Impact**: UX degradation — switching filters intermittently shows an empty roster, making valid archetypes appear to vanish until the user manually clicks back to page 1.
- **Fix sketch**: Reset (or clamp) the page on *any* filter input change, not on a length heuristic — e.g. drive `page` from a `useEffect`/derived reset keyed on the actual filter tuple (`searchTerm, roleFilter, categoryFilter, tierFilter, areaFilter, groupBy`), or always clamp `page = Math.min(page, totalPages - 1)` before slicing so an out-of-range page is impossible regardless of how the list changed.
