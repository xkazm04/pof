## 1. Duplicate state names silently generate uncompilable C++ (duplicate enumerators + redundant cascade)
- **Severity**: high
- **Category**: silent-failure
- **File**: src/components/modules/content/animations/StateMachineEditor.tsx:129-168 (generateEnumCode / generateComputeAnimState); validator gap at src/lib/state-machine-validator.ts:77-179
- **Scenario**: A designer clicks "Add State" twice (each new state is created with the hard-coded `name: 'NewState'`, `flag: 'bIsNewState'` тАФ see `addState` at line 422), or renames two states to the same label, then clicks Export / Copy.
- **Root cause**: Codegen assumes state `name` values are unique, because in UE5 they become C++ enum members (`EARPGAnimState::<name>`). `generateEnumCode` emits one enumerator per state with no de-duplication, and `generateComputeAnimState` emits one `if (flag) return EARPGAnimState::<name>;` per non-default state. The linter (`validateStateMachine`) checks duplicate *priority*, unknown flags, reachability and soft-locks тАФ but never checks for duplicate state names, so nothing warns the user. The UI presents the exported `.cpp` as authoritative ("Generated by PoF").
- **Impact**: data/UX degradation тАФ the exported `.cpp` fails to compile in UE5 (`error: redefinition of enumerator 'NewState'`) and the duplicated identical `if (bIsNewState)` branches are silently dead. The tool produces broken output while reporting success, so the failure only surfaces deep in the engine build.
- **Fix sketch**: Add a `duplicate-state-name` validation rule (mirror the existing `duplicate-priority` check, grouping by `name`) at `error` severity, and have codegen refuse/disambiguate (suffix `_2`) when names collide. Make name-uniqueness an invariant enforced at edit time so this entire class of "valid-looking model тЖТ invalid C++" is impossible.

## 2. Removing the only default state makes ComputeAnimState() fall back to the highest-priority-number state (wrong default animation)
- **Severity**: high
- **Category**: logic-error
- **File**: src/components/modules/content/animations/StateMachineEditor.tsx:143-167 (generateComputeAnimState)
- **Scenario**: A user unchecks "Default / fallback state" on Locomotion (the `StatePropertyEditor` checkbox at line 1296) тАФ e.g. while experimenting тАФ and leaves no state marked `isDefault`. They export the C++.
- **Root cause**: `const defaultState = sorted.find((s) => s.isDefault) ?? sorted[sorted.length - 1];` assumes "no explicit default" is a benign case and picks the *last* state in priority-ascending order тАФ which is the state with the **highest priority number = lowest gameplay priority**. In the default 5-state setup that is `Death`. The generated `ComputeAnimState()` then ends with `return EARPGAnimState::Death;` as the fallback when no flag is set, so an idle character with all flags false snaps to the Death pose. Because every non-default state (now *all* states) also gets an `if (flag) return` branch, the chosen fallback is additionally redundant, but the wrong-fallback return is the real defect.
- **Impact**: corruption of runtime behavior тАФ characters using the exported AnimInstance default to the wrong animation state (e.g. Death) with no error anywhere; a subtle, hard-to-trace gameplay bug shipped from a "successful" export.
- **Fix sketch**: Treat "no default state" as a validation error (require exactly one `isDefault`), or make the fallback deterministic by always synthesizing/selecting the locomotion-typed or lowest-priority-by-intent state, and emit a `// NO DEFAULT STATE DEFINED` guard comment rather than silently choosing the worst candidate.

## 3. Simulation/scan state is not reset when the data source changes, leaving the trace bound to stale node IDs
- **Severity**: medium
- **Category**: state-corruption
- **File**: src/components/modules/content/animations/AnimationStateMachine.tsx:312-326 (displayStates memo) & 360-397 (sim state)
- **Scenario**: A user enters Simulate mode and clicks a few states to build a path (computing `simUnreachable`/`simDeadEnds` against the current `displayStates`). While they trace, the bridge manifest arrives asynchronously (`useManifest` flips `bridgeConnected` true) or a background rescan resolves тАФ so `useBridgeData`/`hasScannedData` change and the `displayStates`/`displayTransitions` memo swaps to a different node set with different IDs (`scanned-<name>` vs `anim-idle` fallbacks vs bridge names).
- **Root cause**: `simPath`, `simUnreachable`, and `simDeadEnds` are independent `useState` that capture node IDs at click time, but nothing invalidates them when the underlying `displayStates` identity changes. The code only clears sim state on explicit Exit/Reset (`toggleSimMode`/`resetSimPath`), assuming the graph is static during a simulation. `validNextStates`/`simEdges` are then computed against the new transitions while `simPath` still holds IDs from the old graph.
- **Impact**: UX degradation / silent incorrectness тАФ the path bar renders raw IDs (`stateMap[id]?.label ?? id`), "unreachable"/"dead-end" overlays become meaningless, and clicks may appear to do nothing because the last path node no longer exists in `displayTransitions`. The designer gets a misleading reachability analysis with no indication it went stale.
- **Fix sketch**: Add a `useEffect` keyed on the data-source signature (e.g. `useBridgeData` + a scan identity/`scannedAt`) that resets `simPath`/`simUnreachable`/`simDeadEnds`, or derive the simulation entirely from `displayStates`/`displayTransitions` so a source change recomputes it instead of leaving orphaned IDs.

## 4. Blender NLA export in the editor swallows failures and reports nothing (success theater)
- **Severity**: medium
- **Category**: silent-failure
- **File**: src/components/modules/content/animations/StateMachineEditor.tsx:357-393 (BlenderNLAExport.handleExport)
- **Scenario**: A user clicks "NLA Export" while Blender is connected but the bridge call fails (Blender busy, script error, socket dropped mid-request).
- **Root cause**: `handleExport` awaits `tryApiFetch<ExecuteOutput>(...)` but discards the returned result object entirely тАФ it never inspects `result.ok` or `result.error`, and only toggles `isExporting` back off in the happy path with no `try/catch`. The design assumes the POST always succeeds (or that failure doesn't matter). This is inconsistent with the sibling export in the same context (`AnimationStateMachine.handleExportToBlenderNLA`, line 429-461) which *does* surface `result.error` and wraps the call in try/catch.
- **Impact**: silent failure тАФ the button returns to its idle label as if the export worked, but no NLA tracks were created in Blender. Worse, if `tryApiFetch` itself rejects (network throw), the un-caught rejection leaves `isExporting` stuck at `true`, permanently disabling the button until remount.
- **Fix sketch**: Mirror the `AnimationStateMachine` pattern тАФ wrap in `try/catch/finally`, check `result.ok`, and render a success/error banner. Better: route all Blender exports through one shared hook that guarantees a result is surfaced, so "fire-and-forget a bridge call" is structurally impossible.
