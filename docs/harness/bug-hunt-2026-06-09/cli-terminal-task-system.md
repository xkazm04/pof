# Bug Hunt — CLI Terminal & Task System
> Total: 4
> Severity: 1 critical, 3 high, 0 medium, 0 low

## 1. Abort closes the UI stream but never kills the spawned Claude process
- **Severity**: critical
- **Category**: resource-leak
- **File**: src/components/cli/useTaskQueue.ts:477-486 (handleAbort); src/lib/claude-terminal/cli-service.ts:333-340 (abortExecution, unused)
- **Scenario**: A user starts a CLI run, sees it going wrong, and hits the Stop button (or Esc). `handleAbort` calls `eventSourceRef.current.close()`, clears the heartbeat, registers the task as failed, and dispatches `ABORT`. The UI instantly looks idle and the user moves on.
- **Root cause**: The abort path operates entirely on the *client* EventSource. The server-side `claude.cmd` child process is keyed only by `executionId`, but `executeTask`/`submitPrompt` discard the `executionId` returned from `/api/claude-terminal/query` (only `streamUrl` and `logFilePath` are read — line 440/463). With no `executionId` retained, the client can never call `DELETE /api/claude-terminal/query?executionId=…`, which is the only thing that invokes `abortExecution` → `process.kill()`. The design assumes "closing the SSE connection stops the work," but the spawn is fully decoupled from the stream subscription.
- **Impact**: Every "aborted" run keeps a real Claude Code process alive on the host — still editing files, still spawning tool subprocesses, and still billing tokens/cost — until it finishes naturally or the **100-minute** `cliExecutionTimeout` fires (cli-service.ts:301-308). Repeated abort-and-retry (the natural user reaction to a bad run) stacks N concurrent orphaned editors mutating the same project, causing runaway cost and file corruption. `abortExecution` is exported but has zero callers, confirming the wire was never connected.
- **Fix sketch**: Capture `executionId` from the query response into a ref; in `handleAbort` (and on unmount while streaming) `await apiFetch('/api/claude-terminal/query?executionId='+id, { method:'DELETE' })` before/after closing the EventSource. Make killing the process — not closing the stream — the definition of abort, so the two can never diverge.

## 2. Fast or output-less runs that exit 0 emit no terminal event — UI hangs until the 100-min safety timeout
- **Severity**: high
- **Category**: silent-failure
- **File**: src/lib/claude-terminal/cli-service.ts:283-291
- **Scenario**: The CLI process exits with code 0 but no `result` stream-json line was parsed (`resultEventEmitted` is false) — e.g. a trivial prompt that completes in under 5s, a run with zero assistant text, or any CLI version/flag combo whose final `result` line doesn't parse. The `close` handler only emits a *synthetic* result when `initEventReceived && assistantMessageCount > 0 && durationMs > 5000`. When any of those is false, it emits **nothing**.
- **Root cause**: Success is signalled only by a `result`/`error` event, but the synthetic fallback is gated behind three heuristics that legitimate successful runs can fail. There is no unconditional "process ended" terminal event. The stream route (stream/route.ts:115-123) and the client (useTaskQueue result/error cases) both treat *only* `result`/`error` as stream-terminal, so a clean exit with none of them leaves every layer waiting.
- **Impact**: The SSE stream stays open (until the 6,100,000 ms safety timeout), `isStreaming` stays true, the heartbeat keeps pinging, and the task never completes — the same "stuck, button disabled forever" failure the abnormal-termination handlers were added to fix, but reached through the *success* path so none of those guards trip. `EventSource.onerror` doesn't fire because the connection is healthy. The whole module's CLI is wedged.
- **Fix sketch**: In the `close` handler, always emit a terminal event when `code === 0 && !resultEventEmitted` (a synthetic `result` with `isError:false`), and emit `error` for any non-zero exit (already done). Make "process closed" — not "a parseable result line appeared" — the invariant that guarantees exactly one terminal event per execution.

## 3. Only the first @@CALLBACK marker is ever parsed; multi-block runs drop their real result
- **Severity**: high
- **Category**: logic-error
- **File**: src/lib/cli-task.ts:116 (CALLBACK_MARKER_RE) & 135-146 (parseCallbackMarker); consumed at src/components/cli/useTaskQueue.ts:328-337
- **Scenario**: On the `result` SSE event, the client runs `extractCallbackPayload(assistantOutputRef.current)`, where `assistantOutputRef` is the concatenation of **every** assistant text chunk for the entire run. If Claude emits more than one `@@CALLBACK:…@@END_CALLBACK` block — a discarded draft and a corrected final, or a re-emit after the prompt nudges it — only one is acted on.
- **Root cause**: `CALLBACK_MARKER_RE` is a non-global regex consumed via `String.prototype.match`, which returns only the **first** match. Worse, the pattern body `[\s\S]*?` is lazy and the closing token is `@@END_CALLBACK`, so given two blocks the match runs from the *first* `@@CALLBACK:` to the *first* `@@END_CALLBACK` — i.e. it always binds to the earliest, typically the draft/throwaway, never the last/authoritative one. The assumption that "there is exactly one marker per run" is unenforced and false in practice.
- **Impact**: The wrong payload (or a stale/abandoned one) is POSTed to the callback URL — checklist items marked complete from a draft, feature-matrix/`/api/catalog` lifecycle transitions written with superseded data, or ability-spec/pipeline state corrupted by the first guess instead of the final answer. Because `staticFields` (moduleId, itemId, catalogId, lifecycle) are merged server-side, the bad payload still passes validation and silently lands in the DB.
- **Fix sketch**: Make the marker regex global and have `parseCallbackMarker` return the **last** valid-JSON marker (iterate all matches, prefer the final one whose body parses). Alternatively scope `assistantOutputRef` to text since the last marker and bind the callback id from the closing block. Either way, "which marker wins" must be deterministic and the latest, not an accident of `.match` returning index 0.

## 4. Late callback settle completes the wrong task after the queue has auto-advanced
- **Severity**: high
- **Category**: race-condition
- **File**: src/components/cli/useTaskQueue.ts:346-355 (Promise.race → currentTaskIdRef) vs 533-545 (queue auto-advance)
- **Scenario**: Task A finishes. The `result` handler dispatches `SSE_RESULT` (synchronously flipping `streaming`→false) and then starts a `Promise.race([cbPromise, timeout(callbackSettleMax=10_000ms)])`. Meanwhile the queue effect sees `streaming===false` and, after `nextTaskDelay=3_000ms`, calls `executeTask(B)`, which sets `currentTaskIdRef.current = B`. When A's callback POST or the 10s timeout settles, the `.finally()` reads `currentTaskIdRef.current` — now **B** — and calls `registerTaskComplete(B, …, !A.isError)` plus `onTaskComplete(B, A.success)`.
- **Root cause**: The completion target is resolved *lazily* from a single mutable ref at settle time, but the settle window (up to 10s, gated on a network POST) is longer than the next-task start delay (3s). The ref is shared across tasks, so a slow callback for A overwrites/aliases B's identity. The code assumes "the current task at settle time is still the task that produced this result," which the auto-advancing queue violates.
- **Impact**: Task B is marked completed/failed before it has actually run (registry record flipped, `onTaskComplete` fires with A's success flag), so B's real terminal event later hits an already-"complete" task — checklist items get marked done for the wrong item, `setSessionRunning(false)` fires mid-run, and feature-fix/scan completion callbacks attribute success to the wrong feature. A's own completion is lost entirely.
- **Fix sketch**: Capture `const tid = currentTaskIdRef.current` at the *top* of the `result` handler (before the async race) and close over that constant in `.finally()`, so completion always names the task that produced the result. Better still, carry the owning `taskId` inside the `ExecutionResult`/SSE event and complete by value, never by reading shared mutable state after an await.
