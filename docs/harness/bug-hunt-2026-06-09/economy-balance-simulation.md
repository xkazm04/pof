# Bug Hunt — Economy & Balance Simulation
> Total: 4
> Severity: 1 critical, 2 high, 1 medium

## 1. Simulation config has no lower bound or NaN guard — `agentCount: 0` / `maxLevel: 0` poisons the entire result with NaN/undefined
- **Severity**: critical
- **Category**: edge-case
- **File**: src/app/api/economy-simulator/route.ts:38-46
- **Scenario**: A client POSTs `{ action: 'simulate', config: { agentCount: 0 } }` (or `agentCount: -5`, `agentCount: 1.5`, `maxLevel: 0`, or a non-numeric string that survives `??`). The route only does `Math.min(body.config?.agentCount ?? 100, 500)` — it clamps the *upper* bound but never the lower bound, never coerces to an integer, and never rejects NaN/negatives. `agentCount: 0` reaches `runSimulation` with an empty agents array.
- **Root cause**: The trust boundary (HTTP POST body) is validated for "too big" but the design assumes the lower end is always sane because the UI's `ConfigField` enforces `min=10`. The server is the real trust boundary and the UI clamp is bypassable (curl, replayed request, `loadRun`'s re-simulate path, or a stored run whose config predates a clamp). With zero agents, `computeMetrics` runs `avgGold = totalGold / agents.length` → `0/0 = NaN`, `medianGold = golds[Math.floor(0/2)] = golds[0] = undefined`, `minGold/maxGold = undefined`, and `gini` is 0. `maxLevel: 0` makes `generateXPCurve(0)` return `[]`, breaking every level-scaled lookup.
- **Impact**: corruption — `NaN`/`undefined` propagate into every metric, alert, drift comparison, and the generated UE5 `EconomyManager.h` (e.g. `ExpectedEndgameGold = undefined`, `InflationThreshold = NaN.f` — uncompilable C++ emitted as "calibrated" output). Charts render with `NaN` heights; saved runs persist garbage baselines that silently corrupt all future drift diffs.
- **Fix sketch**: Validate/clamp at the API boundary with a single helper: coerce each numeric field with `Number()`, reject non-finite values with a 400, and clamp into `[min, max]` (e.g. `clampInt(agentCount, 1, 500)`, `clampInt(maxLevel, 1, 100)`). Make the same helper the single source of truth the UI imports, so client and server agree and "valid" is impossible to forge.

## 2. Client-supplied `flowOverrides` are merged into the engine unvalidated, injecting NaN/negative gold flows
- **Severity**: high
- **Category**: silent-failure
- **File**: src/lib/economy/simulation-engine.ts:244-249 (via route.ts:44)
- **Scenario**: A request includes `config.flowOverrides: [{ id: 'enemy-kill-gold', baseAmount: 'abc' }]` or `[{ id: 'quest-reward', frequencyPerHour: -10 }]` or `[{ levelScaling: NaN }]`. `applyFlowOverrides` does `flows[idx] = { ...flows[idx], ...ov }` — a blind spread with no type/range check on any field — and the override values flow straight into `flow.baseAmount + flow.levelScaling * agent.level` and `Math.floor(flow.frequencyPerHour * eff + rng())`.
- **Root cause**: The override merge trusts that overrides are well-formed `Partial<EconomyFlow>` objects, but they arrive verbatim from `body.config?.flowOverrides` (route.ts:44) with zero validation. A string `baseAmount` makes arithmetic produce `NaN`; a negative `frequencyPerHour` makes `Math.floor(negative + rng())` yield a negative occurrence count so the faucet/sink silently vanishes (and a large negative makes the `for` loop body never run while the modeled flow appears "active"). An override with a different `id` typo is silently ignored (`findIndex` returns -1), so a designer's intended change is dropped with no error.
- **Impact**: corruption + silent-failure — one malformed field NaN-poisons every downstream metric (same blast radius as finding #1) with no error surfaced; subtler negative/typo'd values silently produce a misleadingly "healthy" economy report that designers then ship as calibrated C++.
- **Fix sketch**: Sanitize overrides before merge: keep only known numeric keys, coerce with `Number()`, drop non-finite values, clamp to non-negative ranges, and reject (or warn on) overrides whose `id` matches no flow. Centralize so both `runSimulation` and `generateUE5Code`'s `applyFlowOverrides` path get the same guarantees.

## 3. Death-penalty sink uses start-of-hour gold while every other sink uses running gold — and can spend gold the agent never had
- **Severity**: high
- **Category**: logic-error
- **File**: src/lib/economy/simulation-engine.ts:161-181
- **Scenario**: Within one simulated hour, the death-penalty branch computes `penalty = Math.round(agent.gold * 0.05)` from `agent.gold` (the value at the *start* of the hour, since `agent.gold` isn't updated until line 214). Meanwhile the normal-sink loop just below checks affordability against the *running* balance `agent.gold + goldEarned - goldSpent`. The two sink paths disagree about what "current gold" means inside the same tick.
- **Root cause**: The hour simulation defers applying gold deltas until the end (`agent.gold += goldEarned - goldSpent` at line 214) and accumulates into `goldEarned`/`goldSpent` instead. The death-penalty code was written as if `agent.gold` were live, so it (a) ignores gold earned earlier this hour, and (b) has no affordability guard at all — `goldSpent += penalty` can push `goldSpent` above what the agent actually holds, after which line 215's `if (agent.gold < 0) agent.gold = 0` silently clamps to zero, destroying gold the model claims left via a sink. That clamp also makes `outflow` (derived from `totalGoldSpent`) overstate true drains, biasing inflation/deflation alerts.
- **Root cause (assumption)**: "gold is a single mutable field I can read mid-hour" is false here — the hour is computed as a batched delta, so any sink reading `agent.gold` directly is reading stale state.
- **Impact**: corruption (UX degradation of analysis) — net-flow, velocity, and the inflation/deflation alert thresholds are systematically skewed; the negative-clamp hides the inconsistency so it never surfaces as an error, only as subtly wrong balance recommendations.
- **Fix sketch**: Make the death penalty go through the same running-balance affordability check as other sinks (compute against `agent.gold + goldEarned - goldSpent`, cap the penalty at the affordable amount), or restructure the loop to mutate a single live `balance` variable so there is no "start-of-hour vs running" ambiguity for any flow.

## 4. `runItemEconomySim` aggregates rarity/affix/power from *cumulative lifetime* agent counters per level bracket, double-counting and inflating endgame rarity inflation
- **Severity**: medium
- **Category**: logic-error
- **File**: src/lib/economy/item-economy-engine.ts:350-409, 435-441
- **Scenario**: A 25-level / 80-hour run finishes. For each level `lvl`, the bracket stats pull from `agentsAtOrAbove.get(lvl)` and sum `a.rarityCounts[r]` / `a.affixCounts[key]` — but those are *lifetime* per-agent totals accumulated across the whole run (incremented on every drop at every level the agent passed through, line 308/312). An agent that is level 25 is counted in the bracket for level 1, 2, 3, … and 25, each time contributing its *full lifetime* rarity histogram.
- **Root cause**: The bracket model conflates "items seen *while at* level L" with "lifetime items seen by agents who have *reached* level L." Because high-level agents carry the largest lifetime histograms and appear in every lower bracket via `>= lvl`, early brackets are dominated by late-game drops. The `rarityInflation` ratio (early `lvl<=5` brackets vs `lvl>=maxLevel-3` brackets, lines 398-409) is computed from these overlapping cumulative distributions, so the early-vs-endgame comparison is structurally biased — endgame rarity is compared against an "early" sample that is itself contaminated by endgame data.
- **Impact**: UX degradation — the `rarity-inflation`, `rarity-obsolescence`, and `affix-saturation` alerts (the headline balance guardrails this engine exists to produce) fire or stay silent based on a miscomputed distribution. Designers get confident-looking but wrong "rare items 1.8× more common at endgame" numbers. No crash, but the core analytical output is untrustworthy.
- **Fix sketch**: Record rarity/affix observations into the bracket keyed by the agent's level *at the moment of the drop* (push into `brackets.get(agent.level)` inside the drop loop, like power already is), instead of re-deriving per-bracket aggregates from lifetime per-agent counters via `agentsAtOrAbove`. Then each item is counted in exactly one bracket, and early-vs-endgame inflation compares disjoint samples.
