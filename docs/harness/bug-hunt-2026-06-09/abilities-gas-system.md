# Bug Hunt — Abilities & GAS System
> Total: 4
> Severity: 1 critical, 2 high, 1 medium

## 1. Imported scenario with unbounded numeric fields crashes or hangs the GAS balance simulator
- **Severity**: critical
- **Category**: edge-case
- **File**: src/components/modules/core-engine/sub_ability/gas-balance/data.ts:176 (validator) → src/components/modules/core-engine/sub_ability/gas-balance/simulation.ts:39 (runIteration)
- **Scenario**: A user pastes a base64 scenario code (the ImportScenarioModal "PoB-style" exchange) crafted — or accidentally produced — with `count: 1000000000`, or `attackSpeed: 0`, or `armor: -100`. `decodeScenario` runs `isValidSimScenario`, which type-checks every field and rejects non-finite numbers, then reports "Valid Scenario" and applies it. The next "Run" feeds it straight into `runIteration`.
- **Root cause**: The validator's trust-boundary contract is "well-typed and finite," but the simulation's real contract is "physically sane and bounded." `isValidCombatantStats` only asserts `typeof === 'number' && isFinite(...)`; it never enforces `attackSpeed > 0`, `armor >= 0`, `count >= 1 && count <= N`, `iterations <= 10000`, or `maxHealth > 0`. Three concrete failures follow: (a) `runIteration` does `Array.from({ length: e.count }, …)` (simulation.ts:42) — `count: 1e9` allocates a billion-element array and OOM-crashes the tab; (b) `attackSpeed: 0` makes `playerNextAttack = time + (1 / 0) = Infinity` (simulation.ts:67), so after the first hit the player never attacks again, no enemy dies, and every one of the (up to 10,000) iterations runs the full `while (time < 300)` loop at `dt = 0.05` — ~6,000 steps × N enemies × N iterations, freezing the UI thread despite the rAF chunking (each chunk still runs 200 non-terminating fights synchronously); (c) `armor: -100` yields `armor/(armor+100) = -100/0 = -Infinity`, so `armorMitigation` and `effectiveHp = maxHealth/(1 - mitigation)` become `±Infinity`/`NaN` and poison the health report and histogram. The UI even renders "Valid Scenario" — success theater over a payload that will hang or crash the app.
- **Impact**: crash (OOM / unresponsive tab) and DoS; a shared scenario code is enough to freeze any teammate who validates+applies it.
- **Fix sketch**: Make "valid" mean "within the same bounds the editor sliders enforce." Add a per-field range table (the StatInput min/max already encode these: armor 0–200, attackSpeed 0.1–5, count 1–20, iterations 100–10000, maxHealth ≥ 50, etc.) and reject in `isValidCombatantStats`/`isValidSimScenario` any value outside it. Belt-and-braces: clamp `e.count` and guard `1 / attackSpeed` (treat ≤ 0 as "never attacks" without producing `Infinity`) inside `runIteration` so no future caller can wedge the loop.

## 2. Instant (sub-frame) kills are silently recorded as 0 DPS, deflating the DPS distribution and balance grade
- **Severity**: high
- **Category**: logic-error
- **File**: src/components/modules/core-engine/sub_ability/gas-balance/simulation.ts:88 (ttk=time) → :101 (dps guard)
- **Scenario**: A strong player (high baseDamage/attackPower) one-shots the whole pack on the first simulated frame — common for trash packs, and exactly the case a designer runs to confirm "this build clears trash instantly."
- **Root cause**: The loop starts at `time = 0` with `playerNextAttack = 0`, so the player attacks on the very first tick; if that kills all enemies, `allDead` is true and the loop breaks with `time` still `0`. `runIteration` then returns `ttk: 0`. In `computeResults`, DPS is computed as `it.ttk > 0 ? it.totalDamage / it.ttk : 0` (simulation.ts:101). The guard was added to avoid divide-by-zero, but it conflates "no time elapsed" with "no damage": the fastest, highest-DPS iterations get a recorded DPS of **0**. Those zeros flow into `dpsStats.mean` and become `dpsStats.min`, so the strongest builds report the *lowest* DPS floor and a depressed mean — the opposite of the truth. TTK of `0` also enters `ttkStats.min`/percentiles unguarded.
- **Impact**: UX degradation / silent data corruption of the analytics the whole tab exists to produce — designers tune against a DPS curve that systematically under-reports burst builds.
- **Fix sketch**: Don't let an iteration's elapsed time be zero. Either start the kill clock at the first attack interval, or charge a minimum `dt` for any iteration that landed at least one hit (`ttk = Math.max(dt, time)`), and compute DPS from `totalDamage / max(dt, ttk)`. Make 0-damage and 0-time distinguishable so the guard can't silently zero out real results.

## 3. Non-converging fights are silently scored as a 300s loss, skewing survival rate and the health grade
- **Severity**: high
- **Category**: silent-failure
- **File**: src/components/modules/core-engine/sub_ability/gas-balance/simulation.ts:58 (while time<300) → :88 (return)
- **Scenario**: Player and enemies are tuned so neither side can finish the other within 300 s (e.g. high mutual armor, or a tanky enemy the player chips at slower than it regenerates effective HP). Reachable from the normal sliders alone — a high-armor golem pack vs. a low-baseDamage player.
- **Root cause**: The `while (time < 300)` cap is the only exit when no side dies, but the timeout path is indistinguishable from a real result. On timeout, `allDead` is false, so `playerSurvived = playerHp > 0 && allDead` is `false` — the stalemate is recorded as a *player death* (counted against `survivalRate`) even though the player never actually died. Simultaneously `ttk: time` is recorded as `300`, which lands in `ttkStats.mean`/`p90`/`max`. `buildBalanceHealthReport` then reads `survivalRate` and `ttkStats.mean` literally: a balanced-but-grindy fight is reported as "Players almost always die" (critical survival finding) AND "Fights drag on too long," producing an F grade for an encounter that is merely unresolved. The cap is an assumption landmine: "300 s always means the fight ended" is false.
- **Impact**: corruption of the headline balance verdict — designers are told to slash enemy health / buff the player to "fix" a survival problem that doesn't exist, actively de-tuning the encounter.
- **Fix sketch**: Record a third outcome. Add a `timedOut`/`resolved` flag to `SimIterationResult`; exclude timed-out iterations from `survivalRate` and the TTK distribution (or surface them as their own "stalemate %" finding) instead of folding them in as 300 s losses. Drive the health report off resolved fights plus an explicit unresolved count.

## 4. Forge refinement diff misattributes changes when the LLM renames the class
- **Severity**: medium
- **Category**: logic-error
- **File**: src/components/modules/core-engine/sub_ability/forge/AbilityDiff.tsx:126 → src/components/modules/core-engine/sub_ability/forge/ability-diff.ts:111 (diffLines)
- **Scenario**: A designer refines a forged ability with an instruction that legitimately changes identity ("turn this into a channeled beam"), so the LLM returns a new `className` (the prompt explicitly permits this — ability-forge.ts:204). The "What changed" panel diffs `prior.headerCode` vs `next.headerCode` and labels the file `${next.className}.h`.
- **Root cause**: The code diff is a line-level LCS over the two full files, and the header is labeled with only the *new* class name. When the class is renamed, essentially every line that mentions the old vs. new class (the `UCLASS`, constructor, `GENERATED_BODY`, include of `<Class>.generated.h`, every method qualifier) diffs as a delete+add, so a small semantic refinement renders as a near-total rewrite under a filename that no longer matches the left side. The diff's implicit assumption — "prior and next are the same file, lightly edited" — breaks precisely when the refinement is large, which is when an auditable diff matters most. There's no signal that the file was renamed (no "renamed from X" header, and `CodeDiff` shows only `+N/−N` counts), so the "minimal, auditable" promise in the component's own docstring silently fails.
- **Impact**: UX degradation — the refine-audit feature becomes noise exactly when the change is significant, eroding trust in the forge's "auditable rather than opaque swap" guarantee.
- **Fix sketch**: Detect className change up front (`prior.className !== next.className`) and render an explicit "renamed `GA_Old` → `GA_New`" banner; normalize the class identifier (alias old→new) before the LCS so rename-only churn collapses, leaving the genuine behavioral lines as the visible diff. Label the diff with both filenames when they differ.
