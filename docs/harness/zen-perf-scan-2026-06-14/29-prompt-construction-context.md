## 1. Ability Forge prompt bypasses the entire shared context system
- **Severity**: high
- **Lens**: architecture
- **Category**: divergence / SRP
- **File**: src/lib/prompts/ability-forge.ts:209-281
- **Scenario**: Every other module builder routes through `buildProjectContextHeader()` (prompt-context.ts) so it inherits engine paths, build command, dynamic project state, past-error memory, UE pitfalls, the binary-content tripwire, known-asset paths, and wiring requirements. `buildAbilityForgePrompt` is the sole holdout: it hand-rolls `"## Existing Project Context"`, its own `"## Rules"`, and `"## Output Format"` inline (lines 218-280) and never imports `ProjectContext` or `buildProjectContextHeader` at all.
- **Root cause**: The forge was written before/around the shared-header refactor and was never migrated. The codebase even *anticipates* this drift — `auditPromptString` in prompt-builder.ts:259-285 exists specifically to audit hand-rolled prompts and its doc comment names `ability-forge.ts` as the example.
- **Impact**: GAS code generated by the forge gets none of the project's hard-won guardrails: no MSVC/engine path, no build command to self-verify, no error-memory warnings, no `## Wiring Requirements` (so abilities can come back un-granted/un-activated), no known-asset paths (invented content paths). The single richest module (full .h/.cpp generation) is the least grounded.
- **Effort**: 4 · **Value**: 8
- **Fix sketch**: Add `ctx: ProjectContext` to `ForgeInput`. Build via `new PromptBuilder().withProjectContext(ctx, { knownAssetDomains: ['combat'], errorMemory }).withRawTask(task).withWiringRequirements().withRawOutputSchema(jsonShape).withRawBestPractices(rulesBlock).build()`. Keep the tag-registry/few-shot/radar blocks as `withDomainContext` content. Removes ~40 lines of duplicated header/rules and closes the guardrail gap.

## 2. Boilerplate `extraRules` / best-practice strings copy-pasted across every builder
- **Severity**: high
- **Lens**: architecture
- **Category**: duplication
- **File**: src/lib/prompts/audio-scene.ts:12, audio-events.ts:19, level-design.ts:14, menu-flow.ts:30, material-patterns.ts:8, post-process.ts:21, inventory.ts:76, material-configurator.ts:84
- **Scenario**: The literal `'Generate all code files directly — do NOT ask for confirmation.'` (and its near-twin `'Generate the code files directly — do NOT ask for confirmation.'`) appears in ~10 builders. `'Use MetaSounds ...'`, `'Use UE5 Material system best practices.'`, and `'All ... must be UPROPERTY(EditAnywhere, BlueprintReadWrite) ...'` are each repeated 2-3×, sometimes with subtle wording drift ("all code files" vs "the code files").
- **Root cause**: No shared rule vocabulary — each builder inlines its `extraRules` array as raw string literals. There is a single source of truth for the *header*, but not for the *rule fragments* that feed it.
- **Impact**: Editing one rule (or fixing the "all"/"the" inconsistency) means hunting ~10 files; drift is already present. Each duplicated literal is also a fresh allocation per call.
- **Effort**: 3 · **Value**: 6
- **Fix sketch**: Add a `PROMPT_RULES` const map in prompt-context.ts (`GENERATE_DIRECTLY`, `USE_METASOUNDS`, `MATERIAL_UPROPERTY`, …). Builders compose `extraRules: [PROMPT_RULES.GENERATE_DIRECTLY, …]`. One definition, no drift, reused string references.

## 3. O(transitions × nodes) `.find()` lookups inside prompt-assembly map loops
- **Severity**: medium
- **Lens**: performance
- **Category**: algorithmic / repeated work
- **File**: src/lib/prompts/menu-flow.ts:48-49, level-design.ts:142-144, level-design.ts:199-200
- **Scenario**: `menu-flow.ts` resolves each transition's endpoints with `config.screens.find(...)` **twice** per transition (from + to). `level-design.ts` narrative codegen does `doc.rooms.find(...)` twice per connection (lines 143-144); streaming does `config.zones.find(...)` twice per transition (199-200). Each `.find` is O(nodes), so the transition pass is O(transitions × nodes) — and post-process.ts:33 already shows the intended fix (it pre-builds a `Map`).
- **Root cause**: Endpoint resolution wasn't hoisted into an id→node Map; the nested linear scan was the path of least resistance.
- **Impact**: For large diagrams (a 30-zone world with 60 transitions ≈ 3,600 scans) this is wasted work on every prompt rebuild, and prompts are rebuilt on each config edit / re-render. Diagram editors are exactly where node counts grow.
- **Effort**: 2 · **Value**: 5
- **Fix sketch**: Before the `.map`, build `const byId = new Map(config.screens.map(s => [s.id, s]))` (mirroring post-process.ts:33) and replace each `.find` with `byId.get(t.fromId)`. Turns O(n·m) into O(n+m).

## 4. Identical `capitalize` helper duplicated; `material-configurator` repeats `capitalize(surfaceType)` ~16×
- **Severity**: medium
- **Lens**: both
- **Category**: duplication / repeated work
- **File**: src/lib/prompts/level-design.ts:457-459, src/lib/prompts/material-configurator.ts:118-120
- **Scenario**: `function capitalize(s) { return s.charAt(0).toUpperCase() + s.slice(1); }` is defined verbatim in two prompt files. Worse, inside `material-configurator.ts` `filesSection` (lines 48-79) `capitalize(config.surfaceType)` is recomputed ~16 times for the same input, each allocating a new string.
- **Root cause**: No shared string-utils module for the prompt layer; the same one-liner was re-pasted, and the result was never hoisted to a local.
- **Impact**: Low individual cost but it's pure waste: two copies to maintain (drift risk) plus ~16 redundant `charAt/slice/concat` allocations per material prompt build.
- **Effort**: 1 · **Value**: 3
- **Fix sketch**: Export one `capitalize` from a shared `prompts/_format.ts` (or reuse an existing util) and import in both files. In `material-configurator`, hoist `const Surface = capitalize(config.surfaceType)` once and interpolate `Surface`.

## 5. `getRelevantErrors` re-lowercases the same three strings on every keyword iteration
- **Severity**: low
- **Lens**: performance
- **Category**: missing memoization
- **File**: src/lib/error-memory-db.ts:221-229
- **Scenario**: Per scored record, `lowerPattern`/`lowerMessage`/`lowerFix` are computed once (good) but then `kw.toLowerCase()` is recomputed for every record × every keyword inside the inner loop, and the three record fields are scanned with `.includes` for each keyword. Keyword list could grow with richer task descriptions.
- **Root cause**: Task keywords aren't pre-lowercased once before the scoring map; lowering is pushed into the innermost loop.
- **Impact**: Minor — 30 rows × k keywords extra `toLowerCase()` calls per prompt assembly. Not hot today, but it's on the prompt-injection path and trivially avoidable.
- **Effort**: 1 · **Value**: 2
- **Fix sketch**: Hoist `const kws = taskKeywords.map(k => k.toLowerCase())` above the `records.map(...)` and iterate `kws` in the inner loop. One-line change, removes the per-record re-lowercasing.
